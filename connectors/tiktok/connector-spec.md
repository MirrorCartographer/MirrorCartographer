# TikTok Connector Specification

## Connector name

TikTok Field Connector

## Short definition

A governed ingestion layer that turns TikTok references into MC field objects while preserving provenance, privacy, uncertainty, and user agency.

## Why MC needs it

TikTok often functions as a chaotic external memory field: saved videos, emotional hooks, animal-care ideas, music fragments, product references, research leads, humor, aesthetics, and identity signals all collapse into one feed. MC needs a connector that separates:

- what the TikTok item actually contains,
- what the user noticed,
- what MC infers,
- what becomes useful for a project,
- and what should not be persisted.

## Core object types

### 1. Source object

The canonical link and platform metadata.

Required fields:

- canonical_url
- platform
- creator_handle
- caption
- hashtags
- published_at
- collected_at
- source_access_method
- source_status

### 2. User note object

What the user says the item means or why it matters.

Required fields:

- note_text
- reason_saved
- user_tags
- emotional_signal
- project_relevance
- privacy_level

### 3. MC interpretation object

What MC extracts, clearly labeled as interpretation.

Required fields:

- symbolic_tags
- sensory_tags
- practical_tags
- creative_tags
- research_tags
- confidence
- claim_status
- needs_user_review

### 4. Project link object

Where the item belongs in the GitHub mind or MC system.

Possible links:

- Mirror Cartographer architecture
- animal-care observation
- visual style reference
- music/video reference
- opportunity research
- biology/embodiment idea
- product wedge
- source index
- discard pile

## Allowed connector actions

- Import a user-pasted TikTok URL as a source object.
- Import user-supplied TikTok data export rows.
- Normalize and deduplicate TikTok URLs.
- Store public-safe metadata.
- Store private notes only when explicitly marked private.
- Generate MC tags with uncertainty labels.
- Add public-source URLs to the source index when appropriate.
- Produce a project map from saved content.

## Blocked connector actions

- Scrape behind login walls.
- Download or republish copyrighted video/audio.
- Infer protected attributes or mental health states from viewing behavior.
- Treat likes, saves, or views as direct evidence of identity.
- Create public artifacts from private saved content without abstraction.
- Claim TikTok API data is complete.

## URL normalization rule

Canonical TikTok URL should remove tracking parameters such as:

- `utm_source`
- `utm_medium`
- `utm_campaign`
- `tt_from`
- `is_from_webapp`
- `sender_device`
- other analytics parameters

Short links should be resolved only through a safe, explicit resolver step when permitted.

## Claim labels

Every extracted item must carry one claim label:

- source fact: directly present in TikTok metadata
- user statement: supplied by user
- system inference: generated by MC
- project hypothesis: useful but unproven
- unknown: insufficient context

## Privacy labels

Every item must carry one privacy label:

- public source
- private user note
- public-safe abstraction
- sensitive/private do not publish
- delete/discard

## Minimal working version

A first implementation can be manual:

1. User pastes a TikTok URL and optional note.
2. Connector stores the URL and note.
3. MC asks why it was saved.
4. MC creates tags and claim labels.
5. User approves memory status.
6. Item enters the field map.

## Stronger version

A later version can use OAuth/API access where available and permitted:

1. User authorizes TikTok access.
2. Connector imports permitted metadata.
3. Items are deduplicated.
4. User reviews batches.
5. MC organizes items into project clusters.
6. Only approved abstractions become public GitHub artifacts.
