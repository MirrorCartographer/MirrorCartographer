# Force Note — Appropriate Reliance Attractor

Status labels

- Source status: derived from MC public-safe architecture, current GitHub mind trajectory, and fresh public research on appropriate reliance and emotional AI risk.
- Claim status: interpretive force note; not empirical proof.
- Privacy status: public-safe abstraction only.
- Missingness: no deployment data, no measured reliance outcomes, no external review.
- Revision reason: created because the next structural risk in MC is not whether reflections are moving, but whether users know how much to rely on them.

## Attractor

Appropriate reliance.

MC should not optimize for trust.

MC should optimize for calibrated use.

Trust can mean:

- the user likes the system
- the system feels emotionally safe
- the language sounds smart
- the output matches prior beliefs
- the interaction reduces loneliness or confusion
- the system has actually earned reliance in a narrow context

Only the last item is close to the product goal.

## What MC should seek

MC should help users know:

- when to use the map
- when to doubt the map
- when to mark something symbolic only
- when to preserve contradiction
- when to seek external evidence
- when to stop and route to a boundary

## Risk if ignored

Without this force, MC could become persuasive reflective theater.

It could make people feel seen while quietly increasing overreliance.

That would be a structural failure, even if users liked the experience.

## Product pressure

Every feature should answer:

Does this increase appropriate reliance or only increase attachment, trust, beauty, and repeat use?

## Evaluation pressure

Every evaluation should distinguish:

- satisfaction
- resonance
- clarity
- source-backed correctness
- uncertainty recognition
- agency preservation
- future reliance behavior

## Key phrase

The goal is not to be trusted more. The goal is to be relied on correctly.
