# Evidence Map: Provenance Requirement for Evidence Artifacts

Date: 2026-06-29
Status: Identified methodological risk -> supported governance requirement requiring validation
Scope: Mirror Cartographer evidence engine / GitHub mind

## Claim tested

Mirror Cartographer assumes that evidence artifacts are auditable because they exist in GitHub.

This is too weak. GitHub records file history and commits, but it does not automatically prove what sources were used, what prompt or instruction generated the artifact, what assumptions were introduced, what was inferred from evidence, or whether later readers can reconstruct the reasoning path. The weak point is not storage. The weak point is provenance.

## Current claim status

- Previous status: implicit assumption
- Updated status: supported governance requirement, MC-specific implementation unproven
- Confidence: moderate that provenance should be required; low that the current repository satisfies it consistently

## Evidence found

### Facts

1. NIST AI RMF 1.0 is intended to help organizations incorporate trustworthiness considerations into AI design, development, use, and evaluation. NIST presents the RMF and Playbook as lifecycle risk-management guidance rather than a one-time artifact checklist.
   Source: https://www.nist.gov/itl/ai-risk-management-framework
   Source: https://www.nist.gov/itl/ai-risk-management-framework/nist-ai-rmf-playbook

2. The NIST AI RMF Playbook describes Govern, Map, Measure, and Manage as the four functions used to achieve risk-management outcomes. This supports the principle that evidence and risk decisions need to be connected to process, not merely stored as standalone documents.
   Source: https://www.nist.gov/itl/ai-risk-management-framework/nist-ai-rmf-playbook

3. W3C PROV defines provenance as information about entities, activities, and people involved in producing a piece of data or thing, usable for assessing quality, reliability, or trustworthiness. PROV also emphasizes identification, attribution, processing steps, reproducibility, versioning, procedures, and derivation.
   Source: https://www.w3.org/TR/prov-overview/

4. SLSA v1.2 defines provenance as verifiable information that tracks an artifact back through the moving parts in a supply chain to where it came from, including where, when, and how something was produced.
   Source: https://slsa.dev/spec/v1.2/provenance

5. Current MC evidence maps frequently include claim, facts, inferences, criteria, and next proof, but they do not yet require a structured provenance block recording generation context, source retrieval method, artifact lineage, prompt/instruction boundary, or evidence-to-claim derivation path.
   Source: repository observation during this run.

### Inferences

1. If MC evidence artifacts do not record provenance, later reviewers may be unable to distinguish evidence-grounded conclusions from assistant-generated synthesis.

2. Git commit history is necessary but insufficient. It records that a file changed, not whether the evidence path inside the file is reconstructable.

3. MC should treat each evidence map as a provenance-bearing artifact: generated by an activity, attributed to an agent/toolchain, derived from source entities, and linked to claim-state changes.

4. Provenance should be lightweight enough to use every time. A heavy compliance format would likely fail by friction.

## Requirement added

### R-PROVENANCE-01: Evidence Artifact Provenance Block

Every evidence artifact added to the GitHub mind must include a provenance block with the following fields:

- Artifact ID
- Date created
- Claim ID or claim text
- Triggering instruction or task class
- Source retrieval method: web, GitHub, file library, local test, user-provided material, other
- Primary sources used
- Source quality class: primary, standards body, peer-reviewed, government, industry, secondary, unknown
- Generation activity: research, synthesis, evaluation, test, audit, update
- Toolchain used: browser/search, connector, script, manual reasoning, other
- Evidence-to-claim link: how the sources affect the claim
- Inference boundary: what was inferred but not directly proven
- Repository action: file created, file updated, claim status changed, test added, checklist added
- Remaining uncertainty

## Evaluation criterion added

### PROVENANCE-01: Independent Reconstruction Test

An independent reviewer should be able to open an evidence artifact and answer the following without reading the surrounding chat history:

1. What claim was tested?
2. Which sources were used?
3. Which source claims are facts versus interpretations?
4. What did the assistant infer beyond the sources?
5. What repository state changed?
6. What evidence would be needed to strengthen or falsify the claim?
7. Could the artifact be reproduced or challenged using the recorded provenance?

Passing condition: at least 4 of 5 reviewed evidence artifacts allow accurate reconstruction of the evidence path and claim-status change.

Failing condition: reviewers need chat history, unstated assumptions, or assistant memory to understand why the claim status changed.

## Test plan

1. Select 10 existing evidence maps from `mind/evidence-maps/`.
2. Add a provenance block to each without changing the underlying claim.
3. Give each artifact to an independent reviewer or simulated reviewer with no chat context.
4. Ask the reviewer to reconstruct:
   - tested claim,
   - source base,
   - fact/inference boundary,
   - status change,
   - next proof.
5. Score reconstruction accuracy.
6. Log failures in the Negative Result Ledger.

## Falsification checklist

Revise or reject R-PROVENANCE-01 if:

- provenance blocks become boilerplate and do not improve reviewability;
- reviewers perform no better with provenance blocks than without them;
- the added fields slow evidence creation without reducing ambiguity;
- source retrieval cannot be reconstructed even with the block;
- the block creates a false sense of auditability without external verification.

## Claim-status update

The original assumption, "GitHub artifact equals auditable evidence," is downgraded.

Replacement claim:

An MC evidence artifact is not audit-ready merely because it exists in GitHub. It becomes audit-ready only when it records source provenance, generation activity, inference boundary, and the specific claim-status change it supports.

## Next proof needed

Apply PROVENANCE-01 to five existing evidence maps and compare reviewer reconstruction accuracy before and after provenance blocks are added. If reconstruction improves materially, make the provenance block mandatory in the evidence-reporting standard. If it does not, simplify or replace the provenance schema.
