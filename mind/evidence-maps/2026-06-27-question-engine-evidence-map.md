# Evidence Map: Question Engine as Uncertainty Reduction

Date: 2026-06-27
Status: claim revised, not proven
Artifact type: evidence map + evaluation criterion + falsification checklist
Public-safety level: public-safe; no personal/private material; no medical/veterinary claims

## Claim tested

Weak claim:

> Mirror Cartographer should use a Question Engine where a single sharp question can move the interface by reducing uncertainty, revealing contradiction, or exposing the hidden assumption.

This is promising, but it was under-evidenced. The risk is turning “good questions” into aesthetic theater: questions that feel deep but do not measurably improve orientation, output quality, user agency, or safety.

## Bottom line

Claim status: **partly supported, needs product-specific validation.**

Research supports the narrower claim that clarifying questions can improve LLM outputs when prompts are ambiguous or underspecified. Evidence also supports measuring question value through information gain / expected value of information and through human usability costs. But the stronger MC-specific claim — that one symbolic question can reliably change user perception or interface state in a useful way — is still an inference and must be tested directly.

## Evidence found

### 1. Clarification can improve task outcomes when ambiguity is real

Evidence: “Clarify When Necessary” frames clarification as three subtasks: deciding when clarification is needed, choosing the question, and using the answer. It proposes intent entropy / uncertainty estimation to decide when asking is worth it, and reports that targeted clarification outperforms random clarification when only a small fraction of examples can be queried.

Use for MC: MC should not ask questions constantly. It should detect uncertainty first, then ask only when the answer can plausibly change the reflection or interface state.

Source: Zhang & Choi, “Clarify When Necessary: Resolving Ambiguity Through Interaction with LMs,” arXiv, 2023. https://arxiv.org/abs/2311.09469

### 2. Clarifying questions can improve user-perceived usefulness and generated documents

Evidence: A 2024 study found that users preferred documents produced after follow-up questions over those produced from the initial request alone. The paper also reports qualitative value in questions that were thought-provoking, open-ended, or offered unique insight rather than merely collecting missing facts.

Use for MC: MC’s question engine should not be a form-filler. It should generate questions that reduce structural uncertainty while also helping users notice what they mean.

Source: Tix, “Follow-Up Questions Improve Documents Generated by Large Language Models,” arXiv, 2024. https://arxiv.org/abs/2407.12017

### 3. Question quality can be modeled as expected usefulness of the answer

Evidence: Rao & Daumé rank clarification questions using an expected value of perfect information framing: a good question is one whose expected answer is useful. Their StackExchange-based dataset and expert evaluation support ranking questions by expected downstream value, not surface impressiveness.

Use for MC: MC should score candidate questions by expected map movement: how likely the answer is to change state, resolve ambiguity, expose a contradiction, or adjust permission/confidence.

Source: Rao & Daumé, “Learning to Ask Good Questions: Ranking Clarification Questions using Neural Expected Value of Perfect Information,” arXiv, 2018. https://arxiv.org/abs/1805.04655

### 4. Good questions must account for human answerability, not only model uncertainty

Evidence: Bıyık et al. show that active reward learning improves when queries optimize a tradeoff between information gain and how easy the question is for humans to answer. Questions that maximize system uncertainty alone can be too hard, costly, or redundant.

Use for MC: A question that is theoretically high-information but emotionally vague, exhausting, or impossible to answer is a bad MC question. MC needs an answerability filter.

Source: Bıyık et al., “Asking Easy Questions: A User-Friendly Approach to Active Reward Learning,” arXiv, 2019. https://arxiv.org/abs/1910.04365

### 5. Recent LLM work continues to target ambiguity detection and clarification

Evidence: 2025 work on clarification in coding assistants and QA agents reports that systems can detect ambiguous/underspecified requests and generate clarification questions that users rate as useful or that improve answer quality. These are task-specific and not direct evidence for symbolic reflection, but they strengthen the general architecture direction.

Sources:
- Darji & Lutellier, “Curiosity by Design: An LLM-based Coding Assistant Asking Clarification Questions,” arXiv, 2025. https://arxiv.org/abs/2507.21285
- Naik et al., “Agent-Based Detection and Resolution of Incompleteness and Ambiguity in Interactions with Large Language Models,” arXiv, 2025. https://arxiv.org/abs/2507.03726

## Fact vs inference

### Facts supported by sources

- Clarifying questions can improve LLM task outcomes when prompts contain ambiguity or incomplete information.
- Question asking should be selective; asking when no clarification is needed adds cost and friction.
- Information gain / expected value of information is a plausible formal lens for ranking questions.
- Human answerability matters; the highest-information question is not automatically the best user-facing question.

### MC-specific inferences

- MC can treat symbolic reflection as an uncertainty-reduction process.
- A single strong question may be better than a long reflective dialogue if it causes visible map movement.
- Interface motion should be tied to the answer’s effect on uncertainty, contradiction, permission, or confidence.
- “Deep-sounding” questions should be penalized if they do not change the map.

These are not yet proven. They need direct MC testing.

## Evaluation criterion added

A candidate MC question passes only if it satisfies all five checks:

1. **Uncertainty target** — The system can name what uncertainty the question is trying to reduce.
2. **Expected map movement** — The answer could change at least one visible state: symbol, relation, confidence, contradiction, permission boundary, or next action.
3. **Answerability** — A normal user can answer without needing expert knowledge, emotional overexposure, or excessive effort.
4. **Non-coercion** — The question does not imply diagnosis, fate, hidden truth, or a preferred answer.
5. **No faux-depth** — If the answer would not change anything, the question is decorative and should be rejected.

## Falsification checklist

The Question Engine hypothesis fails if repeated testing shows any of the following:

- Users rate the questions as interesting but the answers do not improve output usefulness.
- The interface changes regardless of answer content.
- Questions increase confusion, self-doubt, or perceived surveillance.
- The system asks questions even when ambiguity is low.
- Question selection is driven by poetic style rather than expected map movement.
- Users cannot tell why a question was asked.
- The same answer does not produce a traceable change in state.

## Product requirement

Add a `QuestionCandidate` object before asking any reflective question:

```json
{
  "question": "string",
  "uncertainty_target": "string",
  "expected_map_movement": ["symbol", "relation", "confidence", "contradiction", "permission", "next_action"],
  "answerability_risk": "low | medium | high",
  "coercion_risk": "low | medium | high",
  "reason_for_asking": "string",
  "reject_if": "string"
}
```

The UI should optionally show a short reason: “I’m asking because this would clarify X.”

## Prototype test plan

Build a 20-card symbolic prompt set. For each prompt:

1. Generate three candidate questions.
2. Score each candidate using the five checks above.
3. Ask the highest-scoring question.
4. Compare output before and after the answer.
5. Record whether the answer changed the map in a traceable way.

Minimum pass threshold for first prototype:

- 80% of asked questions identify a clear uncertainty target.
- 70% of answers cause a visible and explainable map change.
- 0 questions imply diagnosis, fate, hidden truth, or mind-reading.
- Users can skip the question without losing control of the session.

## Design metaphor

Question Engine = **a lantern, not a spotlight.**

A spotlight performs certainty and pressures the user to stand in one place. A lantern makes nearby structure more visible while preserving darkness, ambiguity, and choice.

## Next proof needed

Run a synthetic evaluation with 20 prompts and 60 candidate questions, then manually label:

- uncertainty target clarity
- expected map movement
- answerability risk
- coercion risk
- actual map movement after answer

Only after that should the Question Engine become a core MC requirement.
