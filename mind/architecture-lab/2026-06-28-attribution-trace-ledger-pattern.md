# Attribution Trace Ledger Pattern

Date: 2026-06-28
Status: architecture pattern / prototype requirement
Public-safety level: personal material abstracted; no private user facts stored here

## Architecture question

How should Mirror Cartographer preserve authorship, memory influence, and interpretive process without turning the interface into a legalistic audit log or a surveillance-heavy memory dashboard?

## Short answer

MC should treat every saved interpretation as a process object, not a final meaning. The durable unit is not `symbol = meaning`; it is an attribution trace showing how the interpretation was formed, which parts came from the user, which parts came from AI inference, which parts were edited or rejected, what memory influenced it, what future memory effect it is allowed to have, and how it can be reversed.

## Why this pattern exists

Recent HCI evidence suggests that users can misremember whether ideas or text came from themselves, from AI, or from mixed human-AI workflows. That matters for MC because symbolic reflection can alter self-understanding, future retrieval, visual emphasis, and memory continuity. A normal memory record is too blunt: it preserves content but not authorship, uncertainty, friction, revision, or influence scope.

MC should therefore record the creative/reflection trace in a way that supports agency, correction, and future interpretability.

## Fact vs inference

### Stronger evidence

- Human-AI co-creation creates attribution ambiguity. Research on source memory in AI-assisted writing found that people can later misremember what they created with AI versus without AI, with mixed workflows especially vulnerable to source confusion.
- Attribution judgments in human-AI co-creation depend on contribution type, initiative, amount, values, and disclosure expectations. This supports the need to record contribution roles, not merely final output.
- NIST AI RMF frames trustworthy AI as a design, development, deployment, use, and evaluation concern, and its Playbook organizes suggested actions across Govern, Map, Measure, and Manage. This supports treating provenance as an architectural feature, not a post-hoc note.
- Current provenance work around C2PA/content credentials shows that provenance is moving toward machine-readable and human-readable disclosure, but research also warns that provenance systems can fail or mislead when treated as complete proof.

### MC-specific inference

- Applying media/content provenance concepts to private symbolic cognition is an inference, not a directly proven HCI result.
- The claim that trace ledgers will improve user agency must be tested inside MC.
- Trace visibility may help some users understand authorship and memory influence, but too much detail may increase cognitive load or make the product feel clinical.

## Useful concepts extracted

### 1. Process object

An interpretation is stored as a sequence of moves:

input -> relevance mask -> AI hypothesis -> user edit -> confidence change -> memory decision -> future influence rule -> rollback handle

### 2. Contribution roles

Each meaningful clause or symbolic element should have an origin label:

- user-originated: directly supplied by the user
- ai-inferred: generated by MC from context, memory, or pattern matching
- user-edited: changed by the user after AI output
- ai-reframed: changed by AI after user approval or correction
- contested: explicitly rejected, softened, quarantined, or marked uncertain
- imported-memory: retrieved from prior memory or long-term profile

### 3. Memory influence disclosure

MC should expose when memory influenced an interpretation. It does not need to show private raw memory in public-safe contexts, but it should show the class of influence:

- no-memory mode
- session-only context
- durable user preference
- symbolic history
- prior correction
- safety/agency constraint
- project-level architecture rule

### 4. Future influence rule

Every saved trace needs a scope:

- never store
- store only this session
- store as draft memory
- store as low-confidence pattern
- store as user-confirmed pattern
- store as project requirement
- store as blocked interpretation / do-not-use-again rule

### 5. Rollback and repair

A trace must include a repair path:

- undo storage
- weaken confidence
- split interpretation into alternatives
- hide from future retrieval
- mark as outdated
- correct attribution
- export trace
- delete trace

## Proposed schema

```yaml
trace_id: string
created_at: ISO-8601
mode: symbolic | neutral | scientific | mixed
artifact_type: interpretation | map_delta | memory_update | visual_map | prompt_response

source_input:
  user_supplied_summary: string
  private_detail_policy: abstracted | session_private | explicit_public_safe
  raw_input_stored: boolean

relevance_mask:
  keep: [string]
  ignore: [string]
  uncertain: [string]
  ask_later: [string]
  never_store: [string]

contribution_map:
  elements:
    - element_id: string
      text_or_symbol: string
      origin: user-originated | ai-inferred | user-edited | ai-reframed | contested | imported-memory
      confidence: low | medium | high
      evidence_basis: direct-user-input | session-context | durable-memory | external-source | analogy | inference
      user_visible: boolean

memory_influence:
  used_memory: boolean
  memory_classes: [session-only, durable-preference, symbolic-history, prior-correction, safety-constraint, project-rule]
  raw_memory_exposed: false
  influence_summary: string

interpretation_effect:
  impact_tier: low | medium | high | blocked
  visual_effect: none | emphasis | fade | split | quarantine | warning
  action_pressure: none | suggestion | recommendation | strong-warning
  persistence: none | session | durable-draft | durable-confirmed

future_influence_rule:
  storage_scope: never | session | draft | low-confidence-pattern | confirmed-pattern | project-requirement | blocked-rule
  retrieval_allowed: boolean
  requires_user_confirmation_before_reuse: boolean
  expires_at: ISO-8601 | null

repair_controls:
  rollback_available: boolean
  available_actions: [undo-storage, weaken-confidence, split, hide-from-retrieval, mark-outdated, correct-attribution, export, delete]
  last_repaired_at: ISO-8601 | null

public_safety:
  contains_private_material: boolean
  public_export_allowed: boolean
  abstraction_notes: string
```

## Interface requirement

The ledger should not appear as a spreadsheet by default. It should appear as a symbolic process ribbon:

1. What you gave
2. What MC noticed
3. What MC guessed
4. What you changed
5. What got stored
6. What it may influence later
7. How to undo or correct it

Advanced users can expand the full trace. Default users see only the living trail.

## Product behavior requirement

MC must not save an identity-adjacent interpretation as durable memory unless the trace includes:

- contribution map
- confidence level
- memory influence disclosure
- future influence rule
- rollback control
- public/private export status

## Evaluation criterion

Attribution Trace Criterion:

A user should be able to inspect any saved interpretation and answer all seven questions:

1. What did I directly say or create?
2. What did MC infer?
3. What prior memory influenced this?
4. What did I edit, accept, reject, or soften?
5. What is MC allowed to remember from this?
6. How might this affect future interpretations?
7. How do I undo, correct, or limit it?

Passing threshold for prototype v0.1: at least 80% of test users can answer at least 6 of 7 questions correctly after using the interface on a mixed human-AI interpretation.

## Prototype plan

Name: `attribution-trace-ledger-v0.1`

Build one small flow:

1. User enters a symbolic scene with 3 to 6 elements.
2. MC creates a relevance mask.
3. MC offers 2 to 4 interpretations.
4. User edits one interpretation.
5. User chooses memory scope.
6. MC displays the process ribbon.
7. User performs one repair action.

Test cases should include:

- pure user-originated meaning
- AI-inferred meaning later rejected
- memory-influenced interpretation
- low-confidence metaphor
- identity-adjacent interpretation requiring friction
- public export requiring abstraction

## Falsification checklist

This pattern fails if:

- users still cannot tell which parts came from them versus AI
- users over-trust the trace as proof of psychological truth
- trace visibility makes symbolic mapping feel like paperwork
- memory influence remains unclear
- rollback does not actually prevent future reuse
- public export leaks private details
- the ledger increases friction without improving correction or agency
