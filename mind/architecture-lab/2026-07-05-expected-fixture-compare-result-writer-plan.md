# Expected Fixture Compare Result Writer Plan

Date: 2026-07-05
Status: architecture note / implementation plan
Public safety: public-safe; no private personal material; no secrets; no absolute runtime paths

## Architecture question

How should MC implement the first `compare-governance-expected-fixtures.mjs` result writer so the result JSON is deterministic, schema-valid, Markdown-summarized, and clearly separate from future provenance subjects?

## Current repo facts

The repository already has three pieces that should be composed rather than duplicated:

1. `mind/schemas/governance.expected-fixture-compare.result.v1.schema.json` defines a public-safe comparison result envelope. It explicitly states that the result records comparison custody and digest facts, and is not SLSA provenance or build-authenticity proof.
2. `tools/lib/stable-governance-output.mjs` already provides deterministic JSON sorting, SHA-256 hex digesting, repository-relative path normalization, Markdown escaping, and public-safety redaction primitives.
3. `mind/fixtures/governance.expected-fixture-pairs.v1.json` declares the first fixture pairs and says update mode is verify-only until an explicit update workflow exists.

One contract mismatch must be resolved before the executable writer lands: the compare-result schema currently enumerates `GHF_PAIR_MATCH`, `GHF_PAIR_DRIFT`, `GHF_MISSING_GENERATED_FILE`, and `GHF_MISSING_EXPECTED_FILE`, while the check registry and manifest use `GHF_PAIR_MATCHED`, `GHF_PAIR_DRIFTED`, `GHF_PAIR_GENERATED_MISSING`, and `GHF_PAIR_EXPECTED_MISSING`. The first implementation should not paper over that mismatch. It should either update the result schema to the registry names or add compatibility aliases deliberately.

## Research basis

- Node's current crypto API exposes `createHash()` and `hash.digest()` as the byte-digest primitive suitable for SHA-256 comparison.
- GitHub Actions workflow commands support job summaries, annotations, and masking; this makes the comparison writer responsible for safe emission boundaries, not just file output.
- JSON Schema draft 2020-12 is the right contract surface for validating the result envelope, but schema validation should guard structure, not substitute for byte-comparison logic.
- SLSA provenance models subjects and digests as provenance facts. MC's compare result should borrow digest vocabulary but must not claim to be provenance.
- Recent workflow-maintenance research reinforces that automation output contracts should be explicit and reviewable, because CI/CD automation creates long-term maintenance load.

## Useful concepts extracted

### 1. Byte custody result, not provenance

The writer should produce a custody result that says: these generated bytes match or do not match these checked-in expected bytes. It should not say who built them, what builder identity was used, or whether the build is authentic. Those are future provenance questions.

### 2. Result writer as reducer

The writer should be a pure-ish reducer over already validated manifest pairs and filesystem byte reads:

- input: validated manifest model, generated root, expected root, fixture bytes, unlisted generated outputs
- output: one stable result object, one stable Markdown summary, one output-record manifest
- no manifest schema validation duplication
- no update/bless mode
- no direct raw fixture-body emission

### 3. Deterministic JSON requires shared stable primitives

The writer should call `stableJson()` for result emission and `sha256Hex()` for digesting byte strings or buffers. It should not use local ad hoc `JSON.stringify()` because object-key order and formatting become part of the public API.

### 4. Markdown is a review surface

`summary.md` is for humans and CI logs. It should include counts, pair IDs, artifact kinds, states, paths, and digest prefixes. It must not include full raw fixture bodies, absolute runtime paths, secrets, or unescaped table cells.

### 5. Checks are the normalized event layer

Every pair state should map to exactly one normalized `GHF_*` check. The result summary counts should be derived from pair states and checks, not manually maintained separately.

## Implementation shape

Proposed file:

`tools/compare-governance-expected-fixtures.mjs`

Proposed helper boundary:

`tools/lib/expected-fixture-compare-result-writer.mjs`

The helper should export:

- `compareExpectedFixturePairs({ manifest, generatedFiles, expectedFiles, unlistedGeneratedOutputs })`
- `buildExpectedFixtureCompareResult({ manifest, pairs, checks, unlistedGeneratedOutputs, generatedAtUtc })`
- `renderExpectedFixtureCompareSummary(result)`
- `writeExpectedFixtureCompareOutputs({ result, outputRoot, writeFile })`

The CLI should do orchestration only:

1. Load and validate the expected-fixture pair manifest.
2. Read only declared generated and expected files.
3. Discover generated outputs under the generated root.
4. Reject generated outputs not declared in the manifest.
5. Build pair results and checks.
6. Write deterministic `result.json` and `summary.md`.
7. Set a non-zero exit code when drift, missing files, unlisted output, invalid manifest, or internal error occurs.

## Result state policy

Priority order should be stable:

1. `failed_manifest_invalid`
2. `failed_internal_error`
3. `failed_unlisted_output`
4. `failed_missing`
5. `failed_drift`
6. `passed`

This prevents ambiguous states when several failures occur in one run.

## Public summary shape

The Markdown summary should include:

- title: `Expected Fixture Compare`
- result state
- manifest id and manifest path
- counts table
- pair table: pair id, artifact kind, state, generated path, expected path, generated SHA-256 prefix, expected SHA-256 prefix
- unlisted generated output table only when non-empty
- note: `This is a byte-comparison result, not SLSA provenance.`

All cells should pass through shared Markdown escaping.

## Acceptance requirements

- Result JSON validates against `governance.expected-fixture-compare.result.v1.schema.json`.
- Result JSON is byte-stable for the same manifest and file bytes, except for explicitly injected `generated_at_utc` if the caller does not freeze it in tests.
- Tests freeze `generated_at_utc` for fixture expectations.
- Raw fixture bodies never appear in result JSON, summary Markdown, annotations, or safe details.
- Absolute runtime paths never appear in result JSON, summary Markdown, annotations, or safe details.
- Check-code naming mismatch is resolved before the writer is considered complete.
- The implementation imports shared stable output helpers rather than duplicating digest, path, JSON, or Markdown logic.
- Update/bless mode is absent.

## Test plan

Minimum tests before CI wiring:

1. matched pair emits matched pair state, equal digests, success check, and `passed` result state.
2. drifted pair emits drift state, both digests, drift check, and `failed_drift` result state.
3. missing generated file emits missing-generated state and `failed_missing`.
4. missing expected file emits missing-expected state and `failed_missing`.
5. unlisted generated output emits unlisted-output check and `failed_unlisted_output`.
6. unsafe absolute path in any public detail is redacted or rejected.
7. summary Markdown escapes pipes, backticks, and control characters.
8. result JSON validates against the compare-result schema.
9. schema/check-code mismatch test fails until the contract is aligned.

## Durable decision

The first executable comparison writer should be a deterministic custody reducer around validated manifest pairs. It should avoid provenance claims, avoid update mode, and centralize all public emission through shared stable-governance-output helpers.
