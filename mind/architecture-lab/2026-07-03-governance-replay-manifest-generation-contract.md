# Governance Replay Manifest Generation Contract

Date: 2026-07-03
Status: proposed implementation contract
Public safety: public-safe; no private user material; no personal identifiers; no raw session content.

## Architecture question

How should MC implement manifest generation inside `tools/replay-governance-canonical-json-fixtures.mjs` so `manifest.json` is created deterministically after replay outputs, verified against schema, and used by GitHub Actions without duplicating file-layout logic?

## Current-source research basis

- GitHub `actions/upload-artifact` documents `artifact-digest` as a SHA-256 digest output, supports retention controls, and warns that hidden files should only be included after validating their contents.
- GitHub workflow commands support annotations and job summaries through the workflow command channel and `GITHUB_STEP_SUMMARY`.
- Node `crypto.createHash()` is the stable primitive for computing SHA-256 file/content digests in the replay tool.
- JSON Schema 2020-12 provides the validation vocabulary used for the manifest envelope, but modern JSON Schema complexity research argues for keeping governance schemas shallow and avoiding unnecessary dynamic behavior.

## Updated understanding

The replay artifact manifest should not be generated by GitHub Actions. It should be generated by the governance replay tool itself after all replay outputs are written. GitHub Actions should only run the tool, upload the produced artifact directory, and surface summaries. This keeps the replay tool as the single owner of file layout, digest policy, public-safety checks, and schema validation.

The manifest is therefore an ingestion boundary, not an upload receipt. GitHub's artifact digest can verify the uploaded archive, but MC still needs a file-level manifest that future dashboards can read without needing GitHub Actions context.

## Design pattern: self-describing replay artifact directory

### Principle

One command owns the artifact directory:

`tools/replay-governance-canonical-json-fixtures.mjs --out mind/.generated/governance-replay/canonical-json`

No workflow, dashboard, or follow-on script should reconstruct the layout from duplicated constants.

### Required output layout

```text
<out>/
  manifest.json
  replay-result.json
  summary.md
  annotations.json
  checks.json
```

### Generation order

1. Run canonical JSON fixture replay.
2. Normalize all checks into stable check records.
3. Write `replay-result.json` as canonical JSON.
4. Write `checks.json` as canonical JSON.
5. Write `annotations.json` as canonical JSON.
6. Write `summary.md` as generated Markdown from JSON state.
7. Compute file digests for every generated file except `manifest.json`.
8. Write `manifest.json.tmp` as canonical JSON.
9. Validate the temporary manifest against `governance.replay.artifact.manifest.v1.schema.json`.
10. Rename `manifest.json.tmp` to `manifest.json` only after schema validation passes.

### Manifest ownership fields

The manifest should include:

- `schema_id`
- `schema_version`
- `artifact_kind`
- `producer.tool`
- `producer.version`
- `generated_at_policy`
- `root`
- `files[]`
- `files[].path`
- `files[].media_type`
- `files[].sha256`
- `files[].bytes`
- `files[].role`
- `public_safety.redaction_policy`
- `public_safety.contains_private_material: false`
- `exit_behavior.expected_exit_code`
- `summary.primary_status`

### Determinism rule

The manifest must not include wall-clock timestamps, absolute local paths, machine hostnames, usernames, access tokens, run URLs, or environment-derived secrets. If a time marker is needed, use a policy value such as `deterministic-build-no-wall-clock` rather than a timestamp.

### Digest rule

Digest each generated file by bytes, not by parsed JSON object. For JSON files, first write canonical JSON with a trailing newline policy decided by `tools/lib/governance-canonical-json.mjs`; then hash the exact bytes on disk. This prevents the manifest from claiming a digest for content that differs from what was actually uploaded.

### Public-safety gate

Before writing `manifest.json`, scan all generated text files for high-risk material:

- email-like strings
- credential prefixes such as `ghp_`, `github_pat_`, `sk-`, `xoxb-`
- absolute home-directory paths
- phone-number-like strings
- freeform private-session markers

If any gate triggers, produce a schema-valid failed replay result, omit upload-only convenience fields, set a stable check code such as `GOVERNANCE_REPLAY_PUBLIC_SAFETY_BLOCK`, and exit with failure.

## Implementation requirements

### Tool API

`tools/replay-governance-canonical-json-fixtures.mjs` should expose these internal functions:

- `resolveArtifactLayout(outDir)`
- `writeCanonicalJson(path, value)`
- `writeGeneratedMarkdown(path, replayResult)`
- `hashFileSha256(path)`
- `buildReplayManifest({ outDir, files, replayResult })`
- `validateReplayManifest(manifest)`
- `scanPublicSafety(paths)`
- `writeManifestAtomically(outDir, manifest)`

### Workflow integration

GitHub Actions should use the manifest rather than recreating paths:

1. Run the replay command with a fixed output directory.
2. Print `summary.md` into `GITHUB_STEP_SUMMARY` if present.
3. Upload the whole output directory with `actions/upload-artifact`.
4. Set `if-no-files-found: error`.
5. Keep `include-hidden-files: false`.
6. Use read-only repository permissions.

The workflow must not generate or patch `manifest.json`.

## Acceptance checks

- Running replay twice against the same fixtures produces byte-identical `manifest.json`.
- `manifest.json` validates against `governance.replay.artifact.manifest.v1.schema.json`.
- Every listed file exists relative to the artifact root.
- Every listed SHA-256 digest matches the exact file bytes.
- `summary.md` is generated from the same replay result referenced by the manifest.
- The tool exits `0` for pass, nonzero for replay failure, and nonzero for schema-invalid manifest.
- Public-safety failure produces a stable check envelope rather than leaking raw offending content.
