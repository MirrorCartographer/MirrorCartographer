# Validated Manifest Byte-Comparison Verifier Contract

Status: durable architecture note
Date: 2026-07-04
Public-safety level: public-safe; no private source material, secrets, local paths, or personal records.

## Architecture question

How should MC design the first byte-comparison verifier around validated expected-fixture manifest pairs so it computes stable SHA-256 digests, emits deterministic `GHF_PAIR_MATCHED` / `GHF_PAIR_DRIFTED` checks, rejects unlisted generated outputs, and writes public-safe summaries without duplicating manifest-loader validation logic?

## Researched sources

- Node.js crypto documentation: `crypto.createHash()` and hash digest behavior for stable SHA-256 byte digests.
- Node.js path documentation: repository-relative path resolution/containment should use normalized resolved paths internally, but public output must stay repository-relative.
- GitHub Actions workflow commands documentation: summaries, annotations, and masking require careful command-channel emission and should not receive untrusted/raw fixture content.
- SLSA provenance v1.0: subject names and digests show the right abstraction: report artifact identity plus digest, not artifact body.
- 2026 agentic workflow-injection research on GitHub Actions: untrusted repository/PR/comment/config data can cross from prompt boundaries into scripts and tool actions, so the verifier must treat generated files and manifests as untrusted declarations until validated.

## Useful concepts extracted

### 1. Byte comparison is a custody check, not a semantic review

The verifier must not interpret fixture contents. It should compare bytes only:

1. read generated bytes
2. read expected bytes
3. compute SHA-256 over each byte buffer
4. compare digest strings and byte lengths
5. emit one normalized check per declared pair

This prevents hidden coupling between fixture meaning and verifier behavior.

### 2. The manifest loader owns declaration validity

The verifier must accept only a loader-returned object, not raw manifest JSON. The loader remains responsible for:

- JSON Schema 2020-12 validation
- check-code registry validation
- generated/expected root containment
- repository-relative path normalization
- pre-fixture-read failure ordering
- absolute-path redaction boundary

The byte verifier must not reimplement those rules. It may assert that the loader result has the expected normalized shape, but any failure there is a caller contract violation, not a second schema system.

### 3. Digest records should follow the SLSA-style subject pattern

Each pair comparison should produce a public-safe subject record:

- `pair_id`
- `artifact_kind`
- `generated_path`
- `expected_path`
- `comparison: byte_sha256`
- `generated_sha256`
- `expected_sha256`
- `generated_size_bytes`
- `expected_size_bytes`
- `matched: boolean`

Do not write fixture bodies, content excerpts, absolute paths, environment variables, usernames, hostnames, timestamps, temp directories, or command output.

### 4. Unlisted generated outputs are a separate failure class

The verifier should scan the generated root for files after manifest validation and compare the discovered repository-relative generated paths against the manifest-declared generated paths.

Rules:

- Every generated output under `generated_root` must be declared in `pairs`.
- Every declared generated output must exist.
- Extra generated outputs produce `GHF_UNLISTED_GENERATED_OUTPUT`.
- Missing generated files produce `GHF_PAIR_GENERATED_MISSING`.
- Missing expected files produce `GHF_PAIR_EXPECTED_MISSING`.
- Digest mismatch produces `GHF_PAIR_DRIFTED`.
- Digest match produces `GHF_PAIR_MATCHED`.

This turns the manifest into a public API surface rather than a partial allowlist.

### 5. Summary output must be check-derived

The Markdown summary should be rendered from normalized checks and digest subject records only. It should not include raw caught error messages unless they have passed `redactPublicText()` / `sanitizeSafeDetails()` from the governance check helper.

Recommended sections:

1. verifier contract version
2. manifest id
3. counts: matched, drifted, missing generated, missing expected, unlisted generated
4. pair table with pair id, artifact kind, generated path, expected path, generated digest prefix, expected digest prefix, status
5. final failure mode: `passed` or `contract_violation`

Digest prefixes may be shown for readability, but the full digest should remain in `result.json`.

## Proposed executable boundary

Create `tools/compare-governance-expected-fixtures.mjs` as a thin CLI over reusable library logic.

Inputs:

- `--manifest mind/fixtures/governance.expected-fixture-pairs.v1.json`
- `--result-out <repo-relative output path>`
- `--summary-out <repo-relative output path>`
- no update/bless mode in this stage

Internal flow:

1. Load manifest through `loadExpectedFixturePairManifest()`.
2. If loader returns checks containing manifest failures, write deterministic result/summary and set failure exit code before reading fixture files.
3. Recursively discover generated files under normalized generated root.
4. Reject discovered generated paths not declared in the manifest.
5. For each pair, read generated/expected bytes only after declaration validation succeeds.
6. Compute SHA-256 and byte length for both sides.
7. Emit exactly one terminal comparison check per pair unless the pair has a missing-file check.
8. Write stable `result.json` through `stable-governance-output.mjs`.
9. Write stable `summary.md` through the same helper.
10. Set `process.exitCode = 1` for any error/contract violation; do not call `process.exit()`.

## Minimal result envelope

The verifier result should extend the existing replay-result idea without changing prior schema names until explicitly versioned.

Recommended shape:

```json
{
  "schema": "governance.expected-fixture-compare.result.v1",
  "manifest_id": "dry-run-governance-replay",
  "comparison": "byte_sha256",
  "status": "passed | contract_violation",
  "counts": {
    "pairs": 0,
    "matched": 0,
    "drifted": 0,
    "missing_generated": 0,
    "missing_expected": 0,
    "unlisted_generated": 0
  },
  "subjects": [],
  "checks": []
}
```

The envelope is intentionally not a provenance attestation. It is a verifier event record that can later be referenced by artifact custody and dashboard ingestion.

## Test plan

Add tests before wiring into CI:

1. matching pair produces `GHF_PAIR_MATCHED`, status `passed`, equal digests, exit code 0.
2. changed generated file produces `GHF_PAIR_DRIFTED`, status `contract_violation`, exit code 1.
3. missing generated file produces `GHF_PAIR_GENERATED_MISSING` before digest comparison.
4. missing expected file produces `GHF_PAIR_EXPECTED_MISSING` before digest comparison.
5. extra generated file produces `GHF_UNLISTED_GENERATED_OUTPUT`.
6. invalid manifest from loader prevents any fixture read.
7. output files are byte-identical across repeated runs.
8. result and summary contain no absolute runtime paths.
9. summary is derived from normalized checks and subject metadata only.
10. no update/bless flag is accepted.

## Implementation decision

The first verifier should be **verify-only** and **manifest-scoped**.

It should not create or update expected fixtures, not infer expected files from directory names, not parse fixture semantics, and not duplicate loader validation. Its entire job is to prove that a declared public API fixture surface is byte-stable.

## Requirements update

Add these requirements to the governance replay roadmap:

- `REQ-GHF-BYTE-001`: compute SHA-256 from byte buffers, not strings.
- `REQ-GHF-BYTE-002`: compare only manifest-declared fixture pairs.
- `REQ-GHF-BYTE-003`: reject every generated output not declared in the manifest.
- `REQ-GHF-BYTE-004`: emit one normalized `GHF_*` check for every pair terminal state.
- `REQ-GHF-BYTE-005`: write deterministic result and summary files with repository-relative paths only.
- `REQ-GHF-BYTE-006`: never include fixture body excerpts in public output.
- `REQ-GHF-BYTE-007`: update/bless mode is out of scope until a separate manual workflow contract exists.
