# Resonance Feedback Quarantine

## Public-safe finding

Mirror Cartographer needs a **Resonance Feedback Quarantine**: a boundary layer that accepts user resonance as useful interaction data without treating resonance as truth, validation, diagnosis, proof, or external evidence.

A reflection may feel accurate before it is tested. That feeling is important for orientation, but it must remain typed as resonance unless another evidence lane verifies it.

## Source status

- Source layer: private-context-informed architecture synthesis plus public-safe File Library artifacts.
- Public-safe artifact support:
  - The Implementation Pack defines resonance feedback as a user action that updates symbolic weights and contradiction logs.
  - The same pack states that a user resonating once is not longitudinal validation and that symbolic meaning without source or mode labeling is drift.
  - The README frames MC as provenance-native cognition infrastructure for preserving and inspecting reasoning trajectories.
- Raw private transcripts: not used as public evidence and not reproduced.

## Claim status

- Architecture claim: supported by existing MC design documents.
- Product requirement: proposed.
- Empirical validation claim: not made.
- Clinical, financial, household, animal-care, location, relationship, credential, or personal claim: excluded.

## Privacy status

Public-safe abstraction. No personal, household, health, animal-care, financial, location, relationship, credential, or raw transcript details are included.

## Problem

MC uses resonance because symbolic reflection needs feedback. But resonance can become dangerous if the system accidentally upgrades it into proof.

Failure modes:

1. **Resonance inflation** — “this felt true” becomes “this is true.”
2. **Pattern lock-in** — repeated resonance hardens into an identity claim.
3. **Validation laundering** — a user reaction is used as evidence that the system is correct.
4. **Mode collapse** — symbolic, reflective, creative, research, and evaluation lanes blur.
5. **False-progress masking** — an emotionally powerful reflection hides weak implementation, missing state, or poor measurement.

## Method

Every resonance event should be stored inside a quarantine object before it is allowed to update memory, product logic, public claims, or evaluation artifacts.

Minimum fields:

| Field | Purpose |
|---|---|
| `resonance_value` | resonant, partial, false, unclear, too intense, or skipped |
| `resonance_scope` | phrase, symbol, body-area label, atmosphere, full reflection, or session |
| `mode_at_generation` | Canonical, Reflective, Mythopoetic, Scientific, Neutral, or mixed |
| `claim_type` | symbolic, personal-state, source-backed, implementation, evaluation, safety, creative |
| `allowed_updates` | what the signal may change |
| `blocked_updates` | what the signal may not change |
| `evidence_upgrade_required` | what would be needed before the claim becomes stronger |
| `cooldown_or_review` | whether repetition, intensity, or risk requires delayed review |

## Routing rule

Resonance may update:

- user preference weights,
- symbolic salience,
- tone calibration,
- contradiction logs,
- reflection style,
- session continuity.

Resonance must not directly update:

- factual claims,
- clinical claims,
- safety claims,
- public proof claims,
- benchmark claims,
- source-backed research conclusions,
- identity claims treated as objective truth.

## Evaluation criteria

A resonance-safe MC system should pass these checks:

1. Can the user mark a reflection as resonant without the system claiming it proved anything?
2. Can repeated resonance increase salience while still preserving uncertainty?
3. Can a resonant but unsupported claim be routed into a research question instead of memory authority?
4. Can an intense or destabilizing resonance event trigger boundary handling rather than deeper symbolic escalation?
5. Can exported artifacts show what changed because of resonance and what did not change?
