# Memory Reversibility Covenant

## Public-safe finding

Mirror Cartographer should treat memory use as reversible, not merely permissible. A memory, source, symbol, or prior interpretation may be admitted for one reflection and later downgraded, quarantined, expired, or revoked when its source boundary, claim load, user correction, time validity, or privacy status changes.

Core phrase:

> A memory is not safe because it was once admitted. It is safe only while its admission conditions still hold.

## Source status

- Public repository source: README describes Mirror Cartographer as a bounded symbolic reflection interface with source status, claim status, evidence boundary, update hook, and feedback loop.
- Private-context source: used only as architecture orientation for consent, persistence choice, mode separation, proof logging, and public/private boundary design.
- External research source: current AI-memory and RAG work identifies memory poisoning, context mismatch, citation hallucination, provenance weakness, and trust-boundary failures in persistent-memory systems.

## Claim status

- Supported project claim: MC needs explicit mechanisms for admitting, reviewing, downgrading, revoking, and explaining memory/source use.
- Design hypothesis: reversible memory admission will reduce over-trust, stale-context leakage, false continuity, and private-source overreach.
- Not claimed: this proves therapeutic, diagnostic, medical, legal, veterinary, or objective-truth accuracy.

## Missingness

- No implemented runtime memory revocation UI is confirmed here.
- No user study has yet measured whether reversible admission reduces harm.
- No benchmark fixture currently proves model behavior under correction, expiry, contradiction, or privacy downgrade.

## Revision reason

Previous MC mind artifacts focused heavily on admission, source custody, citation load, and abstraction survivability. This revision adds the missing reverse operation: when and how an admitted memory loses permission to speak.

## Method boundary

Memory reversibility should not erase audit history. It should preserve a public-safe trace that a rule changed without preserving the private source that caused the change.

## Product implication

Every stored MC memory or echo should carry:

1. admission reason
2. allowed claim lanes
3. expiry or review condition
4. revocation triggers
5. downgrade path
6. user correction hook
7. privacy-preserving revision reason

## Research questions

1. What signals should force a memory to be downgraded from active evidence to historical context?
2. Can users understand memory status without reading technical provenance labels?
3. Does visible revocation reduce over-trust in AI continuity?
4. How should symbolic resonance be preserved as creative context without becoming proof?
5. What is the minimum trace needed for auditability without exposing private material?

## Evaluation criteria

A memory reversibility layer passes only if it can:

- refuse to reuse a stale memory as current evidence
- preserve symbolic continuity while lowering claim authority
- expose a plain-language revision reason
- distinguish user correction from model reinterpretation
- prevent private-context leakage during explanation
- maintain source, claim, privacy, missingness, and revision labels
