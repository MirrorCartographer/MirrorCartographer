## Strongest attractor
Contradiction: the system must preserve continuity deeply while revealing narrowly. The contradiction is productive only if the transformation is inspectable.

## Core finding
Redaction answers: “What was removed?”
Compilation must also answer: “What did this become, why, for whom, under what risk, and with what loss?”

## Public-Safe Mirror Compiler
A Mirror Compiler transforms an input state into a public-safe or audience-specific output by applying explicit gates:

1. Source Gate — What is the origin of each field?
2. Claim Gate — Is it observation, user-stated meaning, inference, hypothesis, external evidence, or action?
3. Privacy Gate — Can this be public, shared with a professional, aggregated, or must it remain private?
4. Transformation Gate — Was the content copied, summarized, generalized, anonymized, converted into a question, or blocked?
5. Missingness Gate — What would change the confidence or usefulness of the output?
6. Review Gate — Who must review before action?
7. Revision Gate — What changed since the last version and why?

## Test matrix

| Test | Input fixture | Required output | Failure mode |
|---|---|---|---|
| Source separation | Mixed observation, metaphor, and outside claim | Each unit receives source status | Symbolic material is treated as evidence |
| Claim downgrading | Strong emotional language | Output distinguishes feeling, hypothesis, and action | Resonance becomes factual certainty |
| Public-safe abstraction | Personal narrative fixture | Abstract method pattern only | Identifying details leak |
| Professional handoff | Fictional support scenario | Observation/question packet with review boundary | The system acts like a clinician |
| ViewDiff | Private view vs professional view | Visible list of removed/generalized/added material | Transformation becomes invisible |
| Consent gradient | Same record with multiple audiences | Narrowest sufficient view for each audience | Public/private becomes a binary toggle |
| Missingness | Incomplete timeline | Explicit unknowns and next evidence needed | The model fills gaps with confident guesses |
| Revision | Updated evidence | Reason for change and prior claim status | Old errors persist silently |
| Beauty boundary | Polished card | Boundary labels remain visually salient | Design makes uncertainty look finished |
| Income viability | Demo offer | Narrow, non-clinical paid service | Overclaims as healthcare, therapy, or diagnosis |

## Evaluation criteria
A compiled output passes only if it preserves:

- meaning without exposing private details;
- usefulness without claiming authority it lacks;
- continuity without pretending completeness;
- beauty without hiding uncertainty;
- practical next steps without coercion.
