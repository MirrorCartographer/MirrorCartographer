# Public Compression Loss Register

## Core finding

Mirror Cartographer needs a **Public Compression Loss Register**.

> A private signal can be removed from a public artifact, but the system must still record what function that removed signal was serving.

## Public-safe abstract

Mirror Cartographer repeatedly converts dense private-context material into public-safe architecture: methods, requirements, source-boundary notes, evaluation criteria, implementation plans, and indexes. Redaction protects privacy, but redaction alone can also destroy the reason a design requirement exists.

The Public Compression Loss Register defines how MC should document the architectural function of removed private context without exposing the context itself. It separates **what was removed** from **what role it played**.

This matters because MC depends on continuity, contradiction preservation, mode boundaries, resonance feedback, and state history. If compression removes the private particulars but does not preserve the abstract function, public artifacts can become safe but unintelligible, generic, or detached from the system need that produced them.

## Revision reason

Prior entries established privacy gates, claim separation, mode boundaries, lineage digests, ghost-state retirement, visual grammar adaptation, and reconstruction-risk review. This entry adds a compression-specific rule: public-safe abstraction must preserve the architectural reason for a removed private signal, not merely delete the private signal.

## Product requirement

Every public-safe MC artifact derived from private-context material should include a compression-loss record with these fields:

1. `source_gravity` — none, low, medium, high, or unknown; indicates how much private context shaped the artifact.
2. `removed_detail_class` — personal, household, health, animal-care, financial, location, relationship, credential, raw transcript, or mixed.
3. `retained_function` — requirement-driver, boundary-example, failure-example, mode-example, evaluation-example, ontology-example, implementation-driver, or research-question-driver.
4. `compression_method` — generalization, category replacement, structural abstraction, deletion, pseudonymous example, schema extraction, or method-only rewrite.
5. `loss_type` — evidence loss, emotional-context loss, sequence loss, source-trace loss, implementation-specificity loss, or none.
6. `public_claim_allowed` — yes, no, or review-required.
7. `reconstruction_risk` — low, medium, high, or unknown.
8. `revision_reason` — concise explanation of why the compressed form changed from its source form.

## Evaluation criteria

A compression pass succeeds when:

- The public artifact contains no private source detail.
- The artifact still explains why the requirement exists.
- The removed private material is represented only by class, not content.
- The retained function is explicit enough to guide implementation.
- The claim status is not upgraded by compression.
- The source boundary remains visible.
- The artifact can be audited without access to the raw transcript.
- External release risk is checked for composite reconstruction.

## Implementation plan

1. Add compression-loss metadata to public-safe research notes, product requirements, and release packets.
2. Update the source/index schema so every derived public artifact can point to a non-sensitive retained function.
3. Add a release checklist item: "Does this artifact preserve architectural function after redaction?"
4. Add an audit query that lists public artifacts with high source gravity but missing retained-function labels.
5. Route any artifact with high source gravity plus medium/high reconstruction risk through manual review before external publication.

## Research questions

- How should MC represent evidence loss when the strongest examples cannot be published?
- Should the system prefer fewer stronger public artifacts over many thin public-safe abstractions?
