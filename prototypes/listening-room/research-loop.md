# Listening Room Research Loop

## Purpose

Turn future research into durable improvements to the Listening Room pattern.

## Loop

1. Scan: find one source or concept relevant to observation, uncertainty, embodied signals, symbolic interface design, privacy-safe memory, animal behavior monitoring, or human-AI reflection.
2. Extract: identify the concept that improves the pipeline.
3. Classify: mark the finding as fact, design inference, product hypothesis, evidence gap, or safety constraint.
4. Implement: update one artifact in this folder or create a new evidence/design note.
5. Test: define the smallest static demo or user-facing question that would falsify or strengthen the update.
6. Index: add the canonical source URL to `source-index/sources.md` once.

## Default research questions

- What UI pattern helps users distinguish observation from interpretation quickly?
- What language makes uncertainty feel supportive instead of bureaucratic?
- How should MC handle repeated ambiguous signals across time?
- When should a signal become memory?
- When should a signal dissolve and not be persisted?
- What should be blocked from public-safe abstraction?
- How can creative metaphors be preserved without becoming false claims?
- How can animal observations be useful without claiming diagnosis or decoded emotion?

## Implementation rule

Research only matters if it changes one of these:

- pipeline layer
- schema field
- UI label
- safety boundary
- evaluation criterion
- demo behavior
- source-index entry

## Output format for automation runs

- Researched:
- Finding:
- Classification:
- Artifact changed:
- Source-index changed:
- Next experiment:

## Evaluation criterion

A non-technical user should be able to look at one Listening Room entry and answer within one minute:

1. What was directly observed?
2. What did the system infer?
3. What is uncertain?
4. What could become memory?
5. What should happen next?
