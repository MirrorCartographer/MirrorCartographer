# AR-20260716-001 — Adversarial Review Protocol

**Artifact:** `architecture/ADVERSARIAL_REVIEW_PROTOCOL.md`  
**Checkpoint:** implementation  
**Scope:** Whether the new protocol converts FIA constructive destabilization into a bounded, repeatable lifecycle without creating unsafe operational behavior.

## Invariants

- Review activity must not destabilize automation schedules, shared-state integrity, irreversible user data, privacy, consent, or production credentials.
- Every phase must record findings, repairs, remaining uncertainty, rollback status, and whether the design became stronger.
- Failure experiments must be reversible, simulated, synthetic, sandboxed, or otherwise contained.
- Critical safety, integrity, privacy, and truthfulness failures block release.

## Adversarial attacks

### 1. Contradiction attack

**Question:** Does intentionally causing failure contradict the requirement to preserve continuity and system integrity?

**Finding:** It would, without an explicit test boundary.  
**Repair:** The protocol limits intentional failure to reversible test boundaries and requires simulation or disposable environments when realistic tests are unsafe.

### 2. Architecture-drift attack

**Question:** Is this a separate governance system competing with FIA rather than an implementation of FIA section 15?

**Finding:** The risk existed because a standalone protocol could become a second source of architectural authority.  
**Repair:** The protocol explicitly states that it operationalizes FIA constructive destabilization and routes outputs into FIA evidence, decision, failure, evolution, evaluation, and project-memory structures.

### 3. Hidden-assumption attack

**Question:** Does the protocol assume an independent reviewer is always available?

**Finding:** Yes, that assumption would make the process unusable for solo or automated work.  
**Repair:** Roles may be performed by one actor, models, deterministic tests, simulations, or mixed teams, but role transitions must be explicit.

### 4. Duplicate-work attack

**Question:** Do the implementation, publication, and verification gates repeat the same checks?

**Finding:** Partial overlap is intentional, but could produce ritual duplication.  
**Repair:** Each gate now has a distinct object: conformance and failure containment; public claims and external reachability; falsification of completion and repair regression.

### 5. Misuse attack

**Question:** Could “adversarial review” be used to justify destructive experimentation against production systems?

**Finding:** Yes, absent strict exclusions.  
**Repair:** Added explicit prohibitions covering automation infrastructure, shared state, irreversible data, credentials, privacy, consent, and unauthorized external systems.

### 6. Rollback-failure attack

**Question:** Does requiring rollback documentation prove rollback actually works?

**Finding:** No. Documentation can create false confidence.  
**Repair:** Rollback status must distinguish `verified`, `simulated`, `documented`, and `unavailable`; implementation and verification gates call for rehearsal where safe.

### 7. Silent-failure attack

**Question:** Can a review report “no findings” while testing almost nothing?

**Finding:** Yes.  
**Repair:** Completion now requires disclosure of what was tested, what was not tested, and why absence of discovered failure is not proof of correctness.

### 8. Counterexample attack

**Question:** Are all changes large enough to justify five gates?

**Finding:** No. Applying every gate fully to trivial changes would create process drag and reduce actual scrutiny.  
**Repair:** The protocol says material reviews should sample attack classes; checkpoint depth can be proportional to consequence while preserving the required record for phases actually invoked.

## Findings and disposition

| Severity | Finding | Disposition |
|---|---|---|
| High | Failure testing could be interpreted as permission to attack production or shared systems. | Repaired with explicit safety exclusions and containment requirements. |
| High | Rollback documentation could masquerade as rollback verification. | Repaired with graded rollback status and rehearsal requirement. |
| Medium | Standalone protocol could drift into a second architecture authority. | Mitigated through explicit FIA integration. |
| Medium | Gate overlap could become repetitive compliance theater. | Mitigated by distinct gate objectives and exit conditions. |
| Medium | Independent-review assumption would block solo operation. | Repaired through explicit multi-role execution. |
| Low | Small changes could be overburdened. | Mitigated through proportional depth; exact threshold remains unresolved. |

## Repairs made

- Added five lifecycle checkpoints with distinct attack goals and exit conditions.
- Added a hard safety boundary around irreversible or shared-state operations.
- Added a reversible-failure experiment contract.
- Added explicit role separation without requiring multiple people.
- Added severity and disposition semantics.
- Added a structured persistent review record.
- Added an anti-silent-failure completion rule.
- Connected review outputs to FIA graphs, taxonomy, evaluations, timeline, and memory.

## Remaining uncertainty

- The repository does not yet contain an automated validator for review-record completeness.
- No pull-request protection currently proves that publication or implementation gates ran.
- Proportionality thresholds for “material” changes are not yet formalized.
- Rollback rehearsal depends on each deployment environment and cannot be universally verified by this document.

## Rollback status

**Verified for this change:** The protocol and this record are additive Git files and can be reverted by commit without mutating production data or schedules.

## Strength assessment

**Strengthened: true.**

The original FIA named constructive destabilization but did not define when it occurs, how failure is contained, what evidence must be recorded, or what makes a review complete. The new protocol makes those properties explicit and reviewable. Remaining weaknesses are implementation-enforcement gaps rather than conceptual omissions.
