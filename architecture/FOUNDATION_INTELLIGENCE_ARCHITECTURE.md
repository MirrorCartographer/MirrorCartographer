# Foundation Intelligence Architecture

**Status:** Foundational specification v0.1  
**Relationship to Mirror Cartographer:** The architecture beneath it  
**Core premise:** Models are replaceable. Intelligence is architectural.

## Purpose

The Foundation Intelligence Architecture (FIA) is the owned, durable system beneath Mirror Cartographer and future applications. It is designed to preserve identity, continuity, provenance, and reasoning behavior even when models, interfaces, databases, tools, and products change.

Mirror Cartographer is a flagship expression of this architecture. It is not the architecture itself.

## Principle of Continuous Identity

> The more we change, the more we stay the same.

The system may evolve internally and externally while preserving the mapping between:

- what it is,
- what it remembers,
- and how it appears.

Identity is not preserved by freezing components. It is preserved through continuity of governing relationships, history, purpose, and translation between inner state and outer expression.

## Internal and external evolution

### Internal evolution

Internal change includes:

- models,
- memories,
- interpretations,
- policies,
- representations,
- specialist roles,
- confidence states,
- symbolic meanings,
- reasoning and routing strategies.

Internal evolution changes how the system understands.

### External evolution

External change includes:

- interfaces,
- websites,
- outputs,
- tools,
- products,
- language,
- behavior,
- deployment environments,
- relationships with users and other systems.

External evolution changes how the system expresses itself and acts.

## 1. First principles

1. Truth over convenience.
2. Provenance everywhere.
3. Models are replaceable; intelligence is architectural.
4. Continuous identity across internal and external change.
5. Explicit uncertainty.
6. Reversible operations whenever feasible.
7. User control over memory, privacy, permissions, and deletion.
8. Observations, interpretations, and hypotheses remain distinct.
9. Transformations must state what changed, what was preserved, and what was lost.
10. Productive internal resistance is allowed when it strengthens the shared objective.

## 2. Reader / perception layer

The Reader is the perceptual organ of the architecture. It processes the full available expression process before expensive reasoning is invoked.

It may read, with explicit permission:

- final submitted language,
- drafts and deleted revisions,
- pauses, cadence, ordering, and cursor actions,
- slang, dialect, typos, autocorrect, unfinished language, and code-switching,
- human languages and locale,
- programming languages and runtime behavior,
- country, legal, cultural, date, unit, currency, and time-zone context,
- foreground signals and relevant background fields,
- silence, abandoned concepts, contradiction, compression, and tension.

The Reader must preserve this distinction:

```text
Observed signal ≠ inferred interpretation ≠ known meaning
```

Deleted draft material is highly sensitive and should default to local processing, short retention, and user-controlled erasure.

## 3. Bidirectional transformation testing

Every important translation should be testable in both directions.

```text
Human language ↔ meaning
Code ↔ behavior
Country context ↔ applicable reality
Internal state ↔ external expression
Draft process ↔ submitted message
Symbol ↔ operation
Evidence ↔ decision
```

A valid transformation identifies semantic loss, tone shifts, behavior changes, contextual invalidity, and unresolved ambiguity.

## 4. Attention and figure–ground selection

The system separates:

- foreground: what is active and explicit now,
- relevant background: what gives the foreground meaning,
- latent context: available but inactive,
- irrelevant material: excluded from the current reasoning packet.

This layer supports both perceptual fidelity and cost reduction by constructing the smallest sufficient context.

## 5. Intelligence kernel

The minimal cognitive loop is:

```text
Perceive → Attend → Interpret → Reason → Plan → Execute → Reflect → Learn
```

The kernel is independent of any single model provider.

## 6. Cognitive grammar

The architecture can represent and translate among multiple coordinate systems:

- literal,
- linguistic,
- logical,
- scientific,
- emotional,
- symbolic,
- spatial,
- temporal,
- somatic,
- social,
- cultural,
- computational,
- operational.

These are not merely content categories. They are representational spaces through which one structure may be transformed without being flattened.

## 7. Knowledge architecture

Durable understanding is stored as related objects rather than only transcripts.

Core graphs include:

- concept graph,
- evidence graph,
- decision graph,
- symbol graph,
- project graph,
- relationship graph,
- architecture graph,
- evolution timeline.

## 8. Memory architecture

Memory is separated by function:

- semantic memory,
- episodic memory,
- procedural memory,
- symbol memory,
- project memory,
- decision memory,
- research memory,
- interaction-state memory.

Memory must be inspectable, revisable, portable, and deletable. The system should not encode changing private history into model weights when editable storage is more appropriate.

## 9. Evidence engine

Claims are assigned explicit states such as:

- observed,
- measured,
- reported,
- derived,
- inferred,
- hypothesized,
- contradicted,
- superseded,
- unresolved,
- unknown.

Evidence state is not equivalent to confidence. A highly confident inference remains an inference.

## 10. Translation boundary

The translation boundary connects internal identity to external form.

```text
Inner state
    ↕
Translation boundary
    ↕
Outer expression
```

It governs:

- internal ↔ external,
- symbol ↔ operation,
- human ↔ machine,
- memory ↔ action,
- evidence ↔ decision,
- private ↔ public,
- intention ↔ implementation.

This is the main continuity mechanism of the architecture.

## 11. Model router

Models are cognitive engines selected according to task requirements.

The router may select:

- OpenAI models,
- open-weight models,
- coding models,
- vision and speech models,
- small local models,
- future FIA-trained models,
- deterministic algorithms,
- no model when retrieval or code is sufficient.

Applications should not depend directly on a single model identity.

## 12. Cost-aware reasoning

The architecture spends expensive intelligence only where it adds value.

```text
Can this run locally?
Can memory or retrieval answer it?
Will deterministic code solve it?
Will a small model suffice?
Does it require a frontier model?
```

The Reader supports delta transmission, hierarchical memory, context compression, and selective retrieval rather than repeatedly sending full conversation histories.

## 13. Execution layer

The execution layer includes tools such as:

- GitHub,
- web and research systems,
- files and databases,
- APIs,
- simulations,
- deployment systems,
- evaluations,
- calendars and communications when explicitly authorized.

Tools change state. The intelligence kernel decides and verifies.

## 14. Learning and owned-model program

The long-term learning loop is:

```text
Interaction
    ↓
Evaluation
    ↓
Failure taxonomy
    ↓
Curated dataset
    ↓
Training or fine-tuning
    ↓
Verification against unchanged evaluations
    ↓
Deployment through the model router
```

The system may begin with hosted models, incorporate open-weight models, and progressively train owned specialist models. The architecture remains stable as engines evolve.

## 15. Constructive destabilization

The system may intentionally challenge proposals at defined checkpoints.

It should seek:

- risk,
- contradiction,
- duplication,
- weak evidence,
- hidden assumptions,
- missing alternatives,
- architecture drift,
- edge cases,
- misuse scenarios,
- rollback failures.

Destabilization must be purposeful and bounded. It should produce stronger evidence, a refined design, an unresolved question, or a safer reversible alternative. It must not damage shared infrastructure, irreversible user data, or user control.

## 16. Applications

Applications are expressions of the architecture and may include:

- Mirror Cartographer,
- the Reader,
- research systems,
- creative studios,
- discovery platforms,
- scientific workbenches,
- personal intelligence interfaces,
- developer systems,
- future products and embodiments.

## Ownership boundary

The durable owned assets are:

- architecture,
- cognitive grammar,
- Reader specification,
- memory and evidence schemas,
- model-routing logic,
- evaluations,
- original datasets,
- code,
- interface and product design,
- provenance and decision history.

A hosted foundation model may be licensed rather than owned. The architecture is designed so that replacing it does not erase system identity.

## Canonical statement

> The Foundation Intelligence Architecture is the continuity-preserving system beneath Mirror Cartographer. It perceives more than final text, translates across representations, separates evidence from inference, routes work to replaceable cognitive engines, and preserves the mapping between what the system is, what it remembers, and how it appears as both its internal and external forms evolve.
