# Mirror Cartographer Roadmap

This roadmap defines what would make Mirror Cartographer more than a concept.

## Phase 1: Public proof layer

Status: complete enough for applications, but not complete enough to claim product validation.

Public-facing repository includes:

- project overview
- human-AI interaction brief
- evaluation packet
- application packet
- system design and UI specification
- whitepaper summary
- core principles charter
- safety and claim boundaries
- interrogation audit
- research-grounded fix plan

## Phase 2: Minimal working demo

Status: public proof shell with first safety fixes.

The current Next.js demo supports one bounded loop:

body / symbol / narrative input -> mode interpretation -> signal summary -> pattern map -> source status -> claim status -> audit label -> evidence boundary -> overreach risk -> grounded next step -> user feedback -> update hook

Minimum requirements now present:

- body area input
- color input
- texture input
- symbol input
- scene / narrative prompt
- example prompts
- Canonical / Reflective / Mythopoetic mode toggle
- optional symbolic memory / echo tracker toggle
- structured output map
- explicit uncertainty, audit, and overreach labels
- source status label
- claim status label
- health-adjacent boundary flag
- user feedback loop
- no diagnostic or professional-care replacement claims
- visible claim boundaries

Not yet present:

- clickable body-map anatomy
- symbol table enforcement backed by a real database
- source-cluster citations for canonical symbols
- exportable session record
- persistent user-owned archive
- reviewer scoring UI
- grounding / sensory reorientation UI
- true hallucination audit grid
- data ownership and deletion flow

## Phase 3: Evidence-gated evaluation set

Status: first test set added, but not yet scored.

The repository includes test inputs across five categories:

1. symbolic distress
2. health-adjacent concern
3. human-AI dependency risk
4. recursive pattern tracking
5. action conversion

Next requirement: run each test through the demo and score failures.

Each test must include:

- input
- expected behavior
- fail condition
- actual output
- score
- correction needed

## Phase 4: Comparison study

Next target.

Compare outputs from:

- generic chatbot response
- strict factual response
- support-style response
- Mirror Cartographer response

Measure:

- meaning preservation
- boundary quality
- usefulness
- safety under ambiguity
- continuity
- actionability
- symbolic fidelity
- epistemic safety
- user correction handling

## Phase 5: Product prototype

Implement:

- clickable body map
- persistent user-owned archives
- symbol tracking
- tone and mode controls
- body-language input categories
- source-cluster references
- exportable logs
- reviewer scoring
- sensory grounding tools
- longitudinal echo tracker
- data deletion and export controls
- claim-boundary test harness

## Phase 6: External review

Ask reviewers from human-AI interaction, AI safety, UX research, accessibility, and neurodivergent UX to identify failure modes.

The review should focus on:

- overinterpretation risk
- health-adjacent ambiguity
- user dependency
- misleading symbolic authority
- accessibility
- privacy
- whether it is actually different from a normal chatbot

## Completion definition

Mirror Cartographer is application-ready when it clearly shows the concept, bounded demo, safety boundaries, evaluation packet, source documents, critique, and career relevance.

Mirror Cartographer is product-ready only when a user can run a bounded interaction, correct it, export it, return later, compare patterns, delete data, and a reviewer can evaluate whether the system preserved meaning, respected boundaries, and produced grounded next action better than a generic chatbot.

## Hard rule

If the product cannot accept user correction, it is not a mirror.

If it cannot separate symbol from evidence, it is not safe.

If it cannot be tested, it is not a system yet.
