# Chat to Public GitHub Extraction Pipeline

Revision note:

- Status: public-safe workflow pipeline.
- Reason: created to operationalize the user's instruction: research everything available in chats/files, but do not publish personal specifics to GitHub.
- Source: chat research protocol, public/private boundary work, proof-gap register, guards/filters register, operational connector atlas, execution hub recovery, and current automation loop.
- Boundary: this pipeline governs public-safe extraction. It does not authorize raw transcript publication.

## Core rule

Use the private or long-context layer for understanding.

Publish the public-safe method, not the private case record.

## Pipeline

### Step 1: Search

Find relevant material using query families:

- old names
- corrections
- failures
- build friction
- privacy boundaries
- cognitive profiles
- interpreter teams
- proof gaps
- source gaps
- public value

### Step 2: Read conversationally

Do not treat the chat as static text only.

Look for:

- sequence
- correction
- emotional pressure
- mode mismatch
- command shape
- assistant drift
- access friction
- category rupture

### Step 3: Classify source status

Use:

- source-backed
- user-backed
- memory-backed
- file-backed
- partial source
- unavailable source
- redacted source

### Step 4: Classify claim status

Use:

- observation
- user report
- symbolic hypothesis
- product requirement
- implementation status
- research provocation
- not enough evidence

### Step 5: Classify privacy status

Use:

- public-safe
- public-safe after abstraction
- private by default
- do not publish raw
- explicit approval needed

### Step 6: Extract public-safe value

Convert raw material into:

- method
- template
- schema
- research question
- product requirement
- failure mode
- evidence gap
- implementation plan
- source-boundary note

### Step 7: Label guard effects

Mark if the public artifact may be:

- privacy-abstracted
- source-partial
- publication-filtered
- public-safe but emotionally compressed
- implementation-not-yet-proven
- hypothesis preserved, source withheld

### Step 8: Route to folder

Use:

- `docs/reference/` for reusable schemas and methods
- `docs/research/` for questions and provocations
- `docs/education/` for learning layers
- `docs/privacy/` for boundary rules
- `docs/long-context/` for source history and archive methods
- `docs/long-context/core/` for internal architecture
- `docs/long-context/subjects/` for subject paths
- `src/` for code implementation

### Step 9: Commit only if useful

A GitHub artifact should add at least one of:

- clearer navigation
- better boundary
- stronger proof discipline
- more public value
- reusable schema
- implementation target
- evaluation criterion
- source-gap tracking

### Step 10: Preserve missingness

If something cannot be published, document the gap without exposing the detail.

Claim status:

Product requirement.

Privacy status:

Public-safe after abstraction.

## Failure modes

### Over-publication

Publishing raw context because it feels powerful.

Correction:

Abstract to method.

### Over-redaction

Removing so much that the artifact becomes empty.

Correction:

Preserve pattern, function, and research question.

### False proof

Treating a public doc as validation.

Correction:

Add proof-gap and implementation-status labels.

### Sanitization bias

Making the work too clean to show why it mattered.

Correction:

Preserve conversational pattern without private specifics.

### Source collapse

Treating memory, chat, public source, generated synthesis, and code as equivalent.

Correction:

Label source status.
