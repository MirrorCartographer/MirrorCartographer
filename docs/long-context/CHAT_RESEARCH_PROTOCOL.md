# Chat Research Protocol

Revision note:

- Status: public-safe method derived from researching old chats and notes.
- Reason: created so Mirror Cartographer can continue researching long-context chats more intelligently instead of only searching for obvious project names.
- Source: old chat logs, library search results, user corrections, accessibility notes, proof-machine logs, project-management instructions, and current GitHub organization work.
- Boundary: this protocol is a method for public-safe synthesis. It does not authorize copying raw private chats into GitHub.

## Purpose

The chats are not only an archive of content.

They are also a record of:

- project evolution
- user corrections
- assistant drift
- repeated needs
- broken workflows
- hidden naming changes
- private/public boundary problems
- accessibility failures
- implementation attempts
- research concepts becoming product requirements

To research the chats better, search for **turning points**, not only nouns.

## Core method

Use this loop:

1. Search for obvious names.
2. Search for old names.
3. Search for user corrections.
4. Search for assistant failure points.
5. Search for repeated commands.
6. Search for privacy-sensitive areas.
7. Search for implementation friction.
8. Search for accessible-language needs.
9. Search for source-status and claim-status shifts.
10. Convert findings into public-safe folders, not raw transcripts.

## What counts as a high-value chat signal

### 1. User correction

A correction is one of the strongest signals.

Examples:

- "that is not what I meant"
- "stop using code boxes"
- "do not delete"
- "make it clear where you do revisions and why"
- "everything is important but not everything is equal"

Why it matters:

Corrections reveal the real operating constraints of the system.

### 2. Naming changes

Mirror Cartographer has used multiple names and subsystem labels.

Search for old names, not only the current public name.

Examples:

- Core Entity
- Origin Intelligence
- Echo Engine
- Inward Architecture
- Tone Engine
- Field Logs
- Recursion Maps
- Symbolic Interpreter Core
- Continuity Artifact

Why it matters:

Old names preserve discarded or hidden architecture.

### 3. Assistant drift

When the assistant answered the wrong question or inflated the system, that is useful evidence.

Why it matters:

Assistant drift shows where Mirror Cartographer needs boundaries, claim labels, and correction loops.

### 4. Build friction

Deployment, coding, Vercel, GitHub, and code-access problems reveal product requirements.

Why it matters:

Friction shows where the system needs better implementation, not more theory.

### 5. Accessibility friction

Read-aloud issues, code-block issues, dense-table issues, and visual-only artifacts reveal interface requirements.

Why it matters:

A reflective system fails if the output cannot be accessed by the user's actual reading/listening workflow.

### 6. Private-specific material

Health, animal-care, family, relationship, finances, location, and raw chat details should be abstracted before public use.

Why it matters:

Public method is valuable. Public personal case records are usually unnecessary.

### 7. Repeated command shape

Repeated instructions show what the system should default to.

Examples:

- build instead of only explaining
- organize instead of deleting
- use old chats but protect privacy
- make it searchable
- make it useful for other people

Why it matters:

Repeated command shape becomes product policy.
