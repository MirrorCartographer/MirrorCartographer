# Public Layer Proof Gap Register

Revision note:

- Status: public-safe proof-gap register.
- Reason: created after the user asked where GitHub and the public layer are still not proof of everything learned, designed, or discovered across the private long-context work.
- Source: old evidence registers, current-state packets, execution hub findings, public/private boundary notes, and ongoing chat research.
- Boundary: this file explains what the public GitHub layer proves, what it does not prove, and what remains missing or private. It does not expose private source details.

## Core claim

The public GitHub is not proof of everything Mirror Cartographer has learned or designed.

It is proof of a public-safe layer:

- organized methods
- product architecture
- research questions
- safety boundaries
- source-boundary discipline
- some implementation artifacts
- public-facing documentation

It is not proof of the full private origin archive, the full chat history, or every design path that shaped the system.

## What GitHub can prove

### 1. Public artifact existence

GitHub can prove that certain files, docs, and code artifacts exist at specific commits.

### 2. Public-safe organization

GitHub can prove that a reference library, research index, education layer, privacy layer, and long-context folder structure have been created.

### 3. Current public claims

GitHub can show how Mirror Cartographer is currently framed for outside readers.

### 4. Some implementation status

GitHub can show current code files, workflows, documentation, and build-related changes.

### 5. Boundary discipline

GitHub can show that the public version labels limits, missingness, privacy risks, source status, and claim status.

## What GitHub cannot fully prove

### 1. Full private origin logic

Some of the strongest design reasons came from private or conversational material that should not be copied publicly.

Public-safe abstraction preserves the method, not the full case record.

### 2. Full chat sequence

A public document may summarize a turn, but it cannot fully prove the emotional timing, correction sequence, or conversational pressure that created it.

### 3. Every abandoned path

Old names, discarded architectures, rejected drafts, broken builds, and partial artifacts may not all be present.

### 4. Full implementation readiness

A design document does not prove a working feature.

A README does not prove product validation.

A prototype does not prove clinical, educational, or commercial effectiveness.

### 5. Research novelty

A research provocation does not prove that an idea is new, original, or scientifically validated.

Novelty requires prior-art search, external review, comparison, and testing.

### 6. User benefit

A system can be conceptually strong without proving that it improves outcomes for users.

Benefit requires evaluation, feedback, and longitudinal evidence.

### 7. Private/public completeness

The public repo intentionally excludes private specifics. That means it is safer but less complete.

## Main gap categories

### Source gap

The source exists somewhere, but the public repo does not include it raw.

### Sequence gap

The idea is documented, but the conversation path that created it is missing.

### Evidence gap

The claim is interesting but not tested.

### Implementation gap

The requirement exists, but the feature is not built.

### Evaluation gap

The feature may exist, but it has not been externally evaluated.

### Privacy gap

The private origin is intentionally hidden to protect the user or household.

### Publication-filter gap

A finding may have been blocked, narrowed, or abstracted by publication safeguards.

## Public proof rule

Public GitHub should not pretend to be the whole truth.

It should be a public, inspectable, privacy-safe index of what can be shared.

## Stronger proof path

To make public proof stronger, add:

- source register entries
- missingness labels
- implementation status labels
- evaluation logs
- test cases
- screenshots or demo recordings
- issue history
- pull requests
- external review
- anonymized user studies
- prior-art comparisons

## What not to do

Do not publish private records just to make the repo feel more complete.

Do not convert private specificity into public spectacle.

Do not let public proof gaps erase the fact that private material shaped the architecture.

Do not treat public absence as design absence.

## Best framing

Mirror Cartographer's public GitHub is not the total archive.

It is the public proof surface.

The deeper system includes:

private origin context + available chat/file memory + public GitHub + implementation artifacts + correction loops + evaluation path
