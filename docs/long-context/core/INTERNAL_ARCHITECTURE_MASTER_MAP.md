# Internal Architecture Master Map

Revision note:

- Status: public-safe master map of the internal Mirror Cartographer architecture.
- Reason: created after continued long-context research recovered the role stack, execution hub, connector atlas, lane registry, cognitive profile system, and proof-boundary layers.
- Source: old core spec, whitepaper architecture, operational connector atlas, lane registry, current-state packets, cognitive-profile/interpreter recovery, and current GitHub public-layer work.
- Boundary: this file maps architecture. It does not claim every component is fully implemented, validated, or publicly proven.

## Core finding

Mirror Cartographer is best understood as a routed architecture, not a single tool.

The operating shape is:

signal -> profile -> interpreter -> mode -> output -> feedback -> correction -> archive -> lane -> proof -> next action

## Layer 1: Signal intake

Purpose:

Capture what enters the system before it is flattened.

Signal types:

- body sensation
- symbol
- color
- texture
- metaphor
- typo or language disruption
- repeated question
- image or visual motif
- emotional pressure
- practical task
- build failure
- public/private concern
- source gap

Boundary:

A signal is not proof. It is material to preserve, weight, and route.

## Layer 2: Cognitive profile

Purpose:

Adapt access without turning the profile into a diagnosis.

Profile dimensions:

- direct or poetic language
- visual or verbal processing
- symbolic or literal preference
- body-first or concept-first entry
- fast command or reflective exploration
- uncertainty tolerance
- evidence preference
- accessibility needs
- privacy sensitivity
- action style

Boundary:

A cognitive profile is an access-and-interpretation aid, not a psychological label.

## Layer 3: Interpreter routing

Purpose:

Choose the role needed for the signal.

Interpreter examples:

- symbolic interpreter
- logical interpreter
- world interpreter
- somatic-signal interpreter
- accessibility interpreter
- privacy interpreter
- evidence interpreter
- build interpreter
- evaluator
- archivist

Boundary:

Interpreters are role functions, not independent minds.

## Layer 4: Mode selection

Purpose:

Control output frame and evidence discipline.

Modes:

- Canonical
- Reflective
- Mythopoetic
- Logical
- Scientific
- Practical
- Public-safe
- Build mode
- Research mode
- Privacy mode

Boundary:

Mode labels should prevent drift. Mythopoetic richness should never be confused with factual authority.

## Layer 5: Output formation

Purpose:

Return something usable.

Output types:

- reflection
- question list
- route map
- source-boundary note
- product requirement
- research question
- threshold policy card
- GitHub doc
- implementation issue
- test case
- exportable session record

Boundary:

A beautiful output is not automatically a useful output.

## Layer 6: Feedback and correction

Purpose:

Let the user correct the system and change the next pass.

Feedback types:

- resonated
- partly resonated
- missed me
- overreached
- grounded me
- made me more confused
- preserve as written
- correct it
- map as possible signal

Boundary:

User correction should outrank assistant confidence.

## Layer 7: Archive and memory surface

Purpose:

Preserve source status, missingness, and revision history.

Archive forms:

- GitHub docs
- source register
- file-library notes
- exported conversation files
- long-context memory
- lane registry
- evidence register
- missing-source register

Boundary:

Memory-like continuity may come from external artifacts, not hidden model possession.

## Layer 8: Lane registry

Purpose:

Route work into domains with proof standards.

Lane examples:

- Mirror Cartographer core
- archive continuity
- tech deploy
- AI safety / model behavior
- accessibility
- reasoning / ARC
- visual-symbolic language
- education
- public proof
- privacy

Boundary:

Meaning can connect lanes. Evidence must stay lane-specific.

## Layer 9: Evidence gate

Purpose:

Prevent coherence from pretending to be proof.

Evidence gate asks:

- What kind of claim is this?
- What source supports it?
- What would test it?
- What is missing?
- Is it implemented, drafted, promised, inferred, or imagined?
- What would change its status?

Boundary:

Generated structure and discovered structure must not be collapsed.

## Layer 10: Execution hub

Purpose:

Turn the architecture into action.

Execution hub functions:

- lane tracking
- proof logging
- next executable unit
- tool-limit awareness
- artifact indexing
- public/private routing
- status updates

Boundary:

A hub can organize work. It cannot bypass platform limits or external permissions.

## Layer 11: Publication surface

Purpose:

Decide what becomes public.

Publication checks:

- public-safe?
- source status visible?
- claim status visible?
- privacy status visible?
- missingness visible?
- guard/filter effect visible?
- implementation status visible?

Boundary:

GitHub is the public proof surface, not the total archive.

## Layer 12: Research feedback loop

Purpose:

Use every run to improve the next research pass.

Loop:

search -> cite -> classify -> connect -> test -> artifact -> index -> next action

Boundary:

Search results are not the same as complete understanding. Conversational and emotional reading still matter.

## Master principle

Everything can show information, but not everything is equal.

Mirror Cartographer should preserve signals, weight them honestly, route them through the right interpreter, and keep public proof separate from private origin context.
