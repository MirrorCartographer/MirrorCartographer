## Core idea

Mirror Cartographer was not only a single reflective chatbot.

The internal architecture repeatedly moved toward a multi-role system:

- cognitive profiles
- adaptive tone logic
- symbolic interpreters
- world interpreters
- companion modes
- tutor modes
- integrator modes
- team-like role assignments
- public and private memory layers

## Original system spine

The old core spec defined Mirror Cartographer as a recursive symbolic cognition interface with:

- ENTRY
- FIELD
- RECURSION
- RETURN

It also used three main modes:

- Canonical
- Reflective
- Mythopoetic

The MVP included:

- symbol submission
- reflection generation
- session memory
- exportable artifact

### Compass

Function:

### Convergence

Integrator.

Function:

- connect inner signal and outer world

## Shared core engine modules

### 1. Symbolic Mapping Engine

Purpose:

Capture symbols, colors, body areas, textures, metaphors, and repeated motifs.

Public-safe rule:

Symbolic mapping is user-backed signal, not proof.

### 2. Tone-Adaptive Reflector

Purpose:

Adapt explanations to cognitive style, language preference, directness, symbolic density, and accessibility needs.

Public-safe rule:

Tone adaptation should improve access without manipulating the user or replacing evidence.

### 3. Contextual World Interpreter

Purpose:

Translate external concepts into user-aligned explanations while preserving accuracy and source boundaries.

Public-safe rule:

Personalized explanation must not distort facts.

### 4. Recursive Growth Tracker

Purpose:

Track changes in symbols, roles, questions, loops, and self-description over time.

Public-safe rule:

Longitudinal pattern recognition is not diagnosis or destiny.

## Cognitive profile model

A cognitive profile is not a diagnosis.

It is an access-and-interpretation profile.

It can describe how a user best receives, tests, and acts on information.

Potential profile dimensions:

- direct versus poetic language
- visual versus verbal processing
- symbolic versus literal preference
- body-first versus concept-first entry
- fast command versus reflective exploration
- sensory sensitivity
- uncertainty tolerance
- need for evidence before interpretation
- preference for images, metaphors, examples, steps, or diagrams
- access needs such as read-aloud compatibility

## Cognitive profile fields

A safe profile should include:

- preferred explanation mode
- overload signals
- grounding signals
- correction phrases
- useful metaphors
- prohibited or disliked formats
- evidence preference
- privacy sensitivity
- action style
- review cadence

## Interpreter types

### Symbolic Interpreter

Reads body-symbol-language input as possible meaning signal.

Boundary:

Never converts resonance into proof.

### Logical Interpreter

Converts symbolic material into mechanisms, tasks, variables, or implementation requirements.

Boundary:

Never erases the original symbolic signal.

### World Interpreter

Explains external subjects through the user's profile while preserving factual boundaries.

Boundary:

Personalization must not become misinformation.

### Somatic-Signal Interpreter

Helps organize body-signal observations.

Boundary:

No diagnosis or treatment authority.

### Accessibility Interpreter

Routes output into formats the user can actually use.

Boundary:

Accessibility is not decoration; it controls whether the system works.

### Privacy Interpreter

Checks whether material belongs in public GitHub, private archive, or redacted summary.

Boundary:

Private vividness is not public necessity.

### Evidence Interpreter

Labels claims by source status, claim status, and proof level.

Boundary:

Meaning can connect lanes; evidence must stay lane-specific.

### Build Interpreter

Turns concepts into files, code, tests, issues, workflows, or implementation queues.

Boundary:

A design is not implemented until it changes a working artifact.

## Team model

A team is a set of interpreters assigned to different functions.

The team should not be treated as literal separate minds.

It is a role architecture for AI-assisted work.

## Suggested MC team roles

### Mirror Companion

Focus:

Internal reflection and symbolic fit.

### Researcher

Focus:

Find sources, recover old threads, compare claims, identify gaps.

### Skeptic / Auditor

Focus:

Find overclaiming, missing evidence, privacy risk, and implementation gaps.

### Builder

Focus:

Turn ideas into GitHub docs, code, workflows, and tests.

### Archivist

Focus:

Preserve source status, missingness, old names, and revision reasons.

### Privacy Steward

Focus:

Keep personal specifics out of the public layer.

### Translator

Focus:

Convert symbolic, poetic, scientific, technical, or plain-language material across modes.

### Evaluator

Focus:

Create tests, rubrics, proof standards, and failure labels.

### World Interpreter

Focus:

Explain external systems in profile-adapted but evidence-bounded ways.

### Accessibility Steward

Focus:

Ensure outputs work with read-aloud, low-energy use, plain navigation, and nonstandard input.

## Internal architecture principle

The internal architecture is not a hidden secret system.

It is a routed role stack:

signal -> profile -> interpreter -> mode -> output -> feedback -> correction -> archive -> next role

## Product requirements

- user-selectable cognitive profiles
- editable profile fields
- interpreter routing
- team-role panels
- mode switching
- source and claim labels
- privacy routing
- accessibility preferences
- correction memory
- interpreter audit logs

## Research questions

1. Do cognitive profiles improve comprehension without creating stereotypes?
2. Can interpreter routing reduce assistant drift?
3. Can a team-role interface make AI collaboration easier to audit?
4. Do users trust outputs more appropriately when the interpreter role is visible?
5. Does profile-adapted explanation improve learning while preserving evidence boundaries?
6. Can privacy and evidence interpreters reduce unsafe publication?

## What not to claim

Do not claim that a cognitive profile is a psychological diagnosis.

Do not claim that role names are literal agents.

Do not claim that the team exists as independent minds.

Do not claim that personalized explanation is always more accurate.

Do not claim that symbolic interpreters can know hidden truth.
