# Chat Findings Batch 02

Revision note:

- Status: public-safe synthesis from another old-chat research pass.
- Reason: created to preserve what the latest chat-search pass found and how it changes the research method.
- Source: library search results from old chat logs, accessibility files, proof-machine logs, whitepaper sections, deployment conversations, and collapse/recursion documents.
- Boundary: this file abstracts the findings. It does not copy raw private transcripts or personal case details.

## Finding 1: user corrections are system requirements

Several old-chat results show the assistant drifting, over-symbolizing, or answering the wrong question, followed by user correction.

Public-safe interpretation:

A correction is not just conversational friction. It is product data.

Product requirement:

Mirror Cartographer should preserve user corrections as high-priority routing signals.

Research question:

Can user correction history improve reflective AI safety and fit?

## Finding 2: build mode and broad mode were already present

Old chat logs distinguished between build-oriented work and broad symbolic architecture work.

Public-safe interpretation:

Mirror Cartographer needs mode awareness not only for output style, but for task execution.

Product requirement:

Add task-mode labels:

- build mode
- research mode
- reflection mode
- privacy mode
- public-proof mode

Research question:

Does explicit mode labeling reduce assistant drift?

## Finding 3: accessibility is not cosmetic

Old accessibility material showed that code blocks, dense tables, file-heavy responses, and visual hierarchy can fail when read aloud.

Public-safe interpretation:

Accessibility constraints should shape the writing and interface architecture from the beginning.

Product requirement:

Every important document or UI feature should have a plain-language navigation layer.

Research question:

Do voice-reader-friendly summaries improve long-context project comprehension?

## Finding 4: deployment friction is part of the archive

Old Vercel/Next.js conversations show that setup failures, broken folders, and deployment restarts shaped the project.

Public-safe interpretation:

Implementation friction is not noise. It shows where the product needs clearer instructions and verification.

Product requirement:

Keep deployment and code-quality docs separate from symbolic theory.

Research question:

Does separating build docs from concept docs improve contributor onboarding?

## Finding 5: early language was powerful but overclaim-prone

Older whitepaper language described Mirror Cartographer as revealing, reflecting, and giving structure to pre-verbal knowing. It also used more mythic language such as oracle, operating system, and reflection ritual.

Public-safe interpretation:

The old language preserved the project's emotional force, but the public repo needs clearer boundaries.

Product requirement:

Keep mythopoetic material, but label it by mode and never let it become factual authority.

Research question:

Can mode-labeled poetic language remain useful without increasing false certainty?

## Finding 6: old public-release urgency created privacy risk

Old chat material included a desire to make everything public quickly.

Public-safe interpretation:

Urgency must be balanced by redaction and abstraction.

Product requirement:

Before any public release, run the publication test from `docs/privacy/PUBLIC_PRIVATE_BOUNDARY.md`.

Research question:

Can AI help transform private long-context material into public-safe research artifacts without exposing the case record?

## Finding 7: threshold policies are a strong architecture layer

Proof-machine style files used policy cards with fields like signal, threshold, action, feedback, wrong action to avoid, and minimum missing evidence.

Public-safe interpretation:

This is one of the strongest operational models in the archive.

Product requirement:

Create threshold policy cards for high-risk or high-confusion domains:

- symbolic interpretation
- animal-care observation
- body-signal observation
- novelty claims
- public release
- privacy redaction
- evidence upgrades

Research question:

Do threshold-policy cards reduce overreaction and underreaction?

## Finding 8: feedback loops make actions self-correcting

Old proof-machine material emphasized that actions need feedback signals and thresholds.

Public-safe interpretation:

A next step is weak unless the system knows what would improve, stabilize, escalate, weaken, or reclassify it.

Product requirement:

Every MC output should eventually include:

- signal
- threshold
- action
- feedback
- wrong action to avoid
- minimum missing evidence

Research question:

Does adding feedback fields make reflective AI more useful and safer?

## Finding 9: collapse/recursion documents are useful as symbolic stress tests

Old collapse/recursion artifacts mapped major theories through symbolic cycles.

Public-safe interpretation:

These documents should be treated as creative-theoretical stress tests, not factual claims about the world.

Product requirement:

Create a category for speculative symbolic experiments with strong boundaries.

Research question:

Can speculative theory maps be useful as creativity tools if clearly separated from evidence claims?

## Finding 10: the old-chat research method must search for failure, not only success

The most useful material is often where the system missed, inflated, broke, repeated itself, or needed correction.

Public-safe interpretation:

Failure points are research assets.

Product requirement:

Create a failure-mode register for old chat findings.

Research question:

Can assistant failure logs become a safer design dataset?
