# Cognitive Profile Discussion Recovery

Revision note:

- Status: public-safe recovery of how cognitive profiles were discussed across Mirror Cartographer work.
- Reason: created after the user asked what other ways cognitive profiles were discussed.
- Source: Universal User Adaptive Logic Layer references, Universal Mirror Engine scaffold references, Mirror/Compass/Convergence model, core module summaries, tone-state interpreter notes, comparison notes about ChatGPT role formalization, and current cognitive profile schema.
- Boundary: this file does not turn profiles into diagnoses or fixed categories. It recovers discussion paths and converts them into editable, non-diagnostic architecture.

## Core recovery

Cognitive profiles were discussed in more than one way.

They were not only "learning styles."

They appeared as:

- access profiles
- interpretation profiles
- tone profiles
- symbolic-density profiles
- trauma-layer / sensitivity profiles
- relational-form profiles
- role-routing profiles
- evidence-ordering profiles
- action-mode profiles
- accessibility profiles
- world-translation profiles

## 1. Universal User Adaptive Logic Layer

Original direction:

Mirror Cartographer should reflect, reshape, and expand for any user across cognition, history, and relational form.

Public-safe interpretation:

The system should adapt to how a user receives and tests information, not force everyone through one reflective style.

Product requirement:

Build editable cognitive profiles with source status and user confirmation.

## 2. Universal Mirror Engine

Original direction:

A broader engine that could expand across cognitive profiles, sensitivity layers, and relational forms.

Public-safe interpretation:

The engine should handle difference without treating difference as defect.

Product requirement:

Allow profile variation across language, symbolism, evidence, relationship to the system, and support needs.

## 3. Tone-state interpreter

Original direction:

The system should generate prompts tailored to user tone and symbolic map.

Public-safe interpretation:

Tone is state information.

Product requirement:

Track tone as session context, not permanent identity.

## 4. Cognitive / poetic / mythic / logical tone toggles

Original direction:

Different modes let the same signal be interpreted through different frames.

Public-safe interpretation:

A user may need poetic language to enter the signal and logical language to act on it.

Product requirement:

Mode switching should preserve the same source while changing the frame.

## 5. Body-first profiles

Original direction:

Some users start with body sensation before language.

Public-safe interpretation:

The body can be an entry point for observation, not proof of cause.

Product requirement:

Support body-map input, sensory fields, location/time/context, and what-not-to-conclude boundaries.

## 6. Image-first profiles

Original direction:

Some users think first in images, symbols, color, atmosphere, cards, or visual fields.

Public-safe interpretation:

Visual input can be a serious cognition pathway.

Product requirement:

Support symbol selection, image-to-system translation, glyph labels, color fields, and alt-text accessibility.

## 7. Language-disruption profiles

Original direction:

Typos, compressed phrasing, slang, dialect, and fragments can carry signal.

Public-safe interpretation:

Nonstandard input should not be automatically corrected away.

Product requirement:

Support raw input preservation, corrected version, typo-signal weighting, and user confirmation.

## 8. Evidence-first profiles

Original direction:

Some users need sources, proof labels, or what-not-to-conclude before symbolic interpretation.

Public-safe interpretation:

Evidence order can change trust and usefulness.

Product requirement:

Allow evidence-first mode with claim status, source status, proof gap, and test route.

## 9. Action-first profiles

Original direction:

Some sessions require action, not explanation.

Public-safe interpretation:

Mode mismatch creates assistant failure.

Product requirement:

Profile should include action style: build immediately, research first, explain first, create artifact, update repository, or schedule follow-up.

## 10. Relational-form profiles

Original direction:

The system discussed different relational forms, support loops, and depth mismatches as architecture.

Public-safe interpretation:

Relationships can teach functional roles without publishing private details.

Product requirement:

Separate functional support pattern from personal case record.

## 11. Accessibility profiles

Original direction:

Read-aloud failures, code blocks, dense outputs, and visual-only artifacts shaped system requirements.

Public-safe interpretation:

Access format is part of cognition.

Product requirement:

Include read-aloud compatibility, plain summary first, alt text, and no essential information trapped in code blocks.

## 12. Privacy profiles

Original direction:

Private material can teach architecture, but the public layer must abstract specifics.

Public-safe interpretation:

Different users may need different publication defaults.

Product requirement:

Profile should include public/private routing and explicit approval thresholds.

## 13. Correction profiles

Original direction:

User correction phrases repeatedly changed the system.

Public-safe interpretation:

Correction is alignment data.

Product requirement:

Store correction phrases and route them to fast behavior change.

## 14. Uncertainty profiles

Original direction:

Some users need a clear next action; others can hold ambiguity.

Public-safe interpretation:

Uncertainty tolerance should be adjustable by topic and state.

Product requirement:

Include low, medium, high, and variable uncertainty tolerance.

## 15. World-translation profiles

Original direction:

Compass was a tutor/world interpreter that explains external subjects through the user's learning style.

Public-safe interpretation:

The system should translate the world without distorting facts.

Product requirement:

World Interpreter uses profile adaptation plus evidence boundary.

## Profile safety rules

A cognitive profile should be:

- editable
- user-confirmed where possible
- source-labeled
- topic-specific when needed
- allowed to change
- privacy-aware
- non-diagnostic
- non-stereotyping

## What not to do

Do not sort people into rigid types.

Do not use trauma, sensitivity, cognition, or language style as an identity cage.

Do not assume symbolic users cannot use evidence.

Do not assume evidence-first users cannot use art.

Do not assume one profile works across every state.

Do not make profiles public without consent.

## Best framing

A cognitive profile is not what a person is.

It is how the system should meet them right now.
