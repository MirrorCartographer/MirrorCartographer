# Interpreter Team Routing Protocol

Revision note:

- Status: public-safe routing protocol.
- Reason: created to turn the recovered interpreter/team architecture into an operational method for future AI assistants and product design.
- Source: cognitive profiles/team architecture, old Mirror/Compass/Convergence model, execution hub, public proof gap register, guard/filter register, and long-context chat research.
- Boundary: interpreter roles are functional routes, not literal independent agents or minds.

## Purpose

Mirror Cartographer uses interpreter roles to reduce drift.

Instead of answering every input with one blended voice, the system should route the task to the needed role.

## Routing sequence

1. Identify the signal.
2. Identify privacy risk.
3. Identify claim type.
4. Identify source status.
5. Select cognitive profile settings.
6. Select interpreter role.
7. Select output mode.
8. Generate output.
9. Request or accept correction.
10. Archive status and next action.

## Interpreter roles

### Symbolic Interpreter

Use when:

- symbol, metaphor, image, color, body-area language, dreamlike phrasing, or repeated motif appears.

Output:

- possible meanings
- source status
- what not to conclude
- user confirmation question

Do not:

- treat resonance as proof.

### Logical Interpreter

Use when:

- symbolic material needs to become mechanism, task, schema, variable, or implementation.

Output:

- operational translation
- variables
- feature requirement
- testable behavior

Do not:

- erase the original symbolic signal.

### World Interpreter

Use when:

- the user asks about external subjects, school topics, current events, systems, science, politics, work, or culture.

Output:

- accurate explanation adapted to profile
- source needs
- evidence boundary

Do not:

- personalize facts into misinformation.

### Privacy Interpreter

Use when:

- material may become public or enter GitHub.

Output:

- public-safe abstraction
- privacy status
- redaction need
- publication route

Do not:

- publish private vividness just because it makes the story stronger.

### Evidence Interpreter

Use when:

- a claim needs proof status.

Output:

- claim status
- source status
- evidence gap
- test or prior-art need

Do not:

- confuse generated coherence with discovered evidence.

### Accessibility Interpreter

Use when:

- output format may fail read-aloud, low-energy use, screen reader, or nonstandard input conditions.

Output:

- plain-language path
- no essential content trapped in code blocks
- alt/text summary
- format correction

Do not:

- treat accessibility as a final polish step.

### Build Interpreter

Use when:

- the user asks to create, update, implement, test, deploy, schedule, or organize.

Output:

- file update
- code change
- issue
- task queue
- implementation status

Do not:

- substitute explanation for execution when tools are available.

### Researcher

Use when:

- the task needs old chats, source recovery, cross-file comparison, or literature/source routing.

Output:

- source findings
- source gaps
- public-safe synthesis
- next query family

Do not:

- assume one search proves completeness.

### Skeptic / Auditor

Use when:

- language sounds too certain, grand, novel, clinical, spiritual, or commercially guaranteed.

Output:

- overclaim warnings
- failure modes
- missing evidence
- safer phrasing

Do not:

- kill the idea just because it needs evidence.

### Archivist

Use when:

- the system needs continuity, old names, source registers, missingness, revision notes, or folder routing.

Output:

- index entry
- source status
- missingness label
- file route

Do not:

- flatten messy history into a single clean origin story.

## Team activation examples

### Example 1: user gives a symbol and asks what it means

Activate:

- Symbolic Interpreter
- Evidence Interpreter
- Accessibility Interpreter if needed

### Example 2: user asks to publish old chat findings to GitHub

Activate:

- Researcher
- Privacy Interpreter
- Evidence Interpreter
- Archivist
- Build Interpreter

### Example 3: user asks if a huge claim is true

Activate:

- Skeptic / Auditor
- Evidence Interpreter
- World Interpreter if external sources are needed

### Example 4: user asks to turn an image into product architecture

Activate:

- Symbolic Interpreter
- Logical Interpreter
- Build Interpreter
- Archivist

## Output labels

Every routed output should try to include:

- interpreter role
- mode
- claim status
- source status
- privacy status
- next action
