# Mirror Cartographer GitHub Mind Learning Loop

Revision note:

- Status: public-safe method / implementation protocol.
- Revision reason: created because repeated research runs need a way to add durable understanding to GitHub without turning private-context material into public transcript, biography, medical record, or raw archive dump.
- Source status: synthesized from available Mirror Cartographer file-library materials, the public proof status ledger, export quarantine protocol, user-state entry spec, operational connector atlas concepts, and current repository documentation.
- Claim status: repository workflow architecture and publication discipline. This is not proof of runtime capability, clinical efficacy, financial value, legal validity, or external validation.
- Privacy status: public-safe abstraction only. No personal, household, health, animal-care, financial, location, relationship, credential, or raw transcript details are included.
- Missingness: no automated ingestion tool, no complete repo inventory, no verified live runtime test suite, and no external user-validation dataset are included here.

## Purpose

The GitHub Mind Learning Loop is the rule for converting recurring research into durable public knowledge.

It answers:

- what should be added to GitHub
- what should stay private
- what should become a product requirement
- what should become an evaluation criterion
- what should become a missingness label
- what should be rejected as unsafe or unverified

## Core principle

Private context may teach the architecture.

Private context should not become the product.

The public repository should receive only:

- abstracted methods
- source-boundary notes
- product requirements
- research questions
- evaluation criteria
- privacy-safe indexes
- implementation plans
- status ledgers
- test protocols
- schemas stripped of private content

## Loop stages

### 1. Gather

Input sources may include:

- public GitHub documentation
- private file-library materials
- saved context
- uploaded project packets
- implementation logs
- prior research outputs
- public web research when current external facts matter

Public output should not include raw private source contents.

### 2. Classify

Every finding receives five labels:

- source status
- claim status
- privacy status
- missingness
- revision reason

Use the proof status ledger vocabulary when possible.

### 3. Abstract

Convert raw material into one of these public-safe forms:

- method
- schema
- status label
- boundary rule
- product requirement
- evaluation criterion
- failure mode
- implementation plan
- research question

### 4. Test for publication safety

Before publishing, ask:

1. Does this expose private identity, household, health, animal-care, financial, location, relationship, credential, or transcript detail?
2. Does it claim something is built when it is only designed?
3. Does it treat symbolic material as empirical proof?
4. Does it make a medical, legal, financial, or scientific claim without proper evidence?
5. Does it depend on a missing or placeholder source?
6. Does it create useful work, or only more language?

If unsafe, do not publish. Convert to a missingness note or private-source boundary.

### 5. Commit one durable unit

Each research run should add at most one or two durable units unless implementation requires more.

Preferred unit types:

- `docs/reference/*` for stable protocols
- `docs/long-context/*` for source-batch syntheses
- `docs/evaluation/*` for tests and scoring rules
- `docs/implementation/*` for build plans
- `docs/indexes/*` for public-safe maps

### 6. Reconcile with the ledger

After adding a durable unit, update or cite the proof status ledger when needed.

The ledger should remain the control surface for:

- built
- documented
- designed
- inferred
- symbolic
- private-only
- blocked
- deprecated
- missing

### 7. Produce a run note

Every run should end with a short public-safe run note:

- what was researched
- what was added
- why it matters
- what remains missing

## What the GitHub Mind is

The GitHub Mind is not consciousness, memory, or authority.

It is a public knowledge scaffold that lets Mirror Cartographer preserve architectural understanding across runs.

It should behave like:

- a map
- a ledger
- a source-boundary index
- a product-requirements archive
- a test-planning surface
- an anti-drift device

## What the GitHub Mind is not

It is not:

- a raw diary
- a private archive
- a health record
- a transcript dump
- a claim of completion
- a replacement for runtime tests
- a substitute for external validation
- a place to preserve exposed secrets

## Claim-control rules

### Built claims

Only use if there is inspectable code, live behavior, or test output.

### Designed claims

Use for specs, schemas, flows, and plans.

### Inferred claims

Use when repeated materials support a pattern but no single source fully proves it.

### Symbolic claims

Use for metaphor, meaning, visual grammar, ritual language, and interaction aesthetics.

### Private-only claims

Use when raw material can inform architecture but should not be exposed.

### Missing claims

Use when a file, implementation, test, source, or proof path is expected but unavailable.

## Failure modes

### Bloat

Too many files are added without execution value.

Correction:

Commit only durable units with clear status labels and next actions.

### Claim inflation

A concept is described as if implemented.

Correction:

Downgrade to designed, inferred, symbolic, or missing.

### Private-source leakage

A public document exposes raw private source details.

Correction:

Remove details and publish only schema, boundary, or generalized method.

### Ghost accumulation

Old files keep influencing the system without clear status.

Correction:

Add source status, missingness, and replacement path.

### Symbolic overreach

Metaphor is treated as evidence.

Correction:

Separate symbolic value from empirical proof.

### Endless research without build

The repo gains language but not executable next actions.

Correction:

Every durable unit must include a build, test, or reconciliation path.

## Evaluation criteria

A research run passes if it:

1. Adds a durable public-safe unit or explicitly explains why no write was appropriate.
2. Does not expose raw private source material.
3. Labels source, claim, privacy, missingness, and revision reason.
4. Advances implementation, testing, indexing, or claim control.
5. Does not repeat existing content without adding a new control surface.
6. Identifies the next executable gap.
