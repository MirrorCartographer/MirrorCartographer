# Public / Private Boundary

Revision note:

- Status: privacy and publication policy.
- Reason: created to protect private personal, household, animal-care, medical, financial, location, and relationship details while keeping Mirror Cartographer useful as a public research and product reference.
- Source: project-management instruction to keep private information out of public GitHub while organizing long-context research.
- Boundary: this policy is practical repo hygiene, not legal advice.

## Public-safe rule

The public GitHub should preserve methods, patterns, product architecture, evaluation logic, research provocations, and anonymized examples.

It should not expose private life details that are unnecessary for understanding the project.

## Keep private by default

Do not publish:

- addresses or exact living locations
- phone numbers or private emails
- financial account details
- private health records
- lab values or appointment timelines
- animal names tied to medical histories
- clinic names tied to private cases
- relationship conflict details
- trauma details
- raw conversation exports
- private screenshots that reveal accounts, locations, or identifiers
- unredacted resumes unless intentionally public

## Public-safe alternatives

Use categories instead of private specifics.

Examples:

- "household animal" instead of a pet name
- "animal-care observation" instead of a specific diagnosis or medication event
- "body-signal observation" instead of private clinical details
- "personal archive" instead of raw chat export
- "saved-media archive" instead of account-specific screenshots
- "public proof artifact" instead of a private application packet

## Meaningful redaction

Redaction is meaningful when it changes privacy exposure, even if the project concept stays the same.

Any redaction should say why it happened.

Example:

Revision note: revised public-safe version to preserve the method while removing private household, health, or animal-care details.

## Safe public examples

Public examples should be:

- anonymized
- generalized
- non-diagnostic
- non-identifying
- useful for product design
- clear about uncertainty

## Private source handling

Old chats, files, and memory can be used to create public-safe indexes and methods.

They should not be copied wholesale into GitHub.

Use:

- summaries
- categories
- search terms
- anonymized patterns
- product requirements
- evaluation questions
- boundary statements

Avoid:

- raw transcripts
- raw health details
- raw animal medical details
- raw relationship details
- raw account screenshots

## Publication test

Before adding a detail to GitHub, ask:

1. Does this help someone understand Mirror Cartographer?
2. Could this expose private life, health, animal, financial, location, or relationship information?
3. Can the same research value be preserved with an anonymized category?
4. Would the user want this public if a stranger, employer, researcher, or scraper read it?
5. Is the detail necessary, or only vivid?

If it is vivid but not necessary, abstract it.

## Repository rule

Public method, private specifics.

The architecture can be public.

The personal case record does not need to be public.
